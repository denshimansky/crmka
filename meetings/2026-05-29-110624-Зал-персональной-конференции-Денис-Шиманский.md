## Основные темы

### 1. Проблема дублей и путаница базы хранения со статусами
В текущей системе база хранения клиента смешана со статусом (пробник, актив, выбывший, потенциал, архив), что породило дубли: один ребёнок мог одновременно числиться в нескольких базах. Анна описала разделение: теперь есть отдельные сущности — лид, контакт/потенциал, клиент (родитель + подопечные), а воронка продаж — это статус внутри этих баз.

### 2. Подготовка к ночной выгрузке данных и обработка дублей
Анна планирует ночью очистить воронку, выгрузить все базы (выбывшие, потенциал и пр.) и столкнётся с большим количеством дублей по номеру телефона. Денис предложил использовать Claude для анализа плоской таблицы и автоматического схлопывания дублей: один родитель — несколько подопечных в строках с одинаковым телефоном.

### 3. Архитектура выгрузки и ТЗ для Саши и Димы
Для корректной миграции нужна унифицированная выгрузка из текущей системы и загрузка в новую CRM. Денис предложил сделать через Сашу кнопку выгрузки с максимальным числом столбцов, а Дима параллельно реализует загрузку в новой CRM. Это должно стать воспроизводимым процессом для всех центров сети.

### 4. Матрица иерархии статусов
Так как один ребёнок может одновременно быть в нескольких базах (выбывший + потенциал + чёрный список), Дима запросил у Анны матрицу приоритетов: какой статус главнее при конфликте. Например: чёрный список > выбывший > потенциал.

### 5. Переплаты и долги
Анна нашла отчёт по остаткам оплаченных занятий (переплаты и долги), однако там только имя ребёнка и сумма — без возможности провалиться в карточку. Проблема усугубляется полными тёзками (Захаров Захар в нескольких центрах). Денис попросил ссылку на отчёт, чтобы попробовать вытащить данные без участия программистов.

---

## Ключевые решения

1. При выгрузке использовать **плоскую структуру**: одна строка = родитель + один подопечный + статус базы. Дубли по телефону — допустимы на этапе выгрузки, схлопывание сделает Claude.
2. Для устранения дублей использовать **Claude**: загрузить плоскую Google-таблицу, задать целевую структуру на втором листе, получить очищенный результат.
3. Анна составляет **матрицу иерархии статусов** (приоритет при конфликте баз) — передаёт Диме как часть ТЗ на загрузку.
4. Запросить у **Саши** разработку унифицированной кнопки выгрузки из текущей системы по общему шаблону.
5. **Оля** параллельно ведёт данные в двух местах (старая и новая система), выявляет косяки; Дима оперативно правит.

## Открытые вопросы

- Как технически корректно выгрузить данные по **переплатам и долгам**: в отчёте нет возможности провалиться в карточку клиента, а тёзки (например, несколько «Захаров Захар» в разных центрах) различаются только по родителю и отчеству.
- Каким образом передать Денису **ссылку/доступ** к отчёту по остаткам оплаченных занятий для анализа без программистов.
- Финальная **инструкция для других центров** по выгрузке и миграции данных должна быть готова к 15 июня — методология ещё не зафиксирована и обкатывается впервые.
