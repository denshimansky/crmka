## Ключевые темы обсуждения

### 1. Партнёрство и роли
- **Денис** — техническая часть, разработка, размещение, сопровождение продукта.
- **Анна** — методология, поиск клиентов, формирование требований, маркетинг, найм при необходимости.
- На старте деньги не нужны — только время обоих. Затраты на серверы, домены, внешние сервисы появятся позже.
- Возможно потребуется помощник (Настя) для поддержки клиентов при переезде.

### 2. Целевые сегменты и типы абонементов
Выделены три типа клиентов с разной бизнес-логикой:
- **Календарный абонемент** (детские центры) — N посещений за период, стоимость = цена занятия x кол-во занятий в месяце. Привязка к конкретному направлению.
- **Пакетный абонемент** (типа Левиты, взрослые студии) — 12/48/96 занятий, без привязки к конкретному занятию, клиент сам записывается на любое из расписания. Срок действия ограничен (30/120/365 дней).
- **Фикс за месяц** (школы, сады) — фиксированная оплата за месяц + поштучное питание с перерасчётом. Календарная система списания питания.

### 3. Сущности и бизнес-логика
- **Клиент = родитель = плательщик.** Ребёнок — информационное поле (имя, возраст, ДР), не участвует в финансовых расчётах.
- **Абонемент** привязан к клиенту-родителю. У одного родителя может быть несколько абонементов (разные дети, разные направления). Всё идёт в один LTV.
- **Убрать понятие "договор"** из системы. Заменить на "абонемент" — предоплата за определённое количество определённых услуг.
- Лид → Заявка → Пробник → Ожидание оплаты → Активный абонемент.
- Статусы лидов: активный лид, потенциал (перезвоните позже), нецелевой, чёрный список, архив.
- Обязательно отдельное поле "дата следующего контакта".

### 4. Финансовый учёт
- **Выручка** (начисленный метод) ≠ **Оборот** (кассовый метод/ДДС). Оба отчёта нужны, они не взаимозаменяемы.
- Расходы делятся на **постоянные** и **переменные** (обязательно флаг/галочка при вводе).
- Постоянные расходы (аренда, интернет, зарплата управленцев) распределяются пропорционально выручке по филиалам/направлениям.
- Амортизация оборудования — расход привязывается к кабинету/филиалу и размазывается на срок гарантии.
- Ввод расходов должен быть максимально простым для админов: сумма → категория → филиал → кабинет/все.

### 5. Боли текущей системы (1С)
- Кривая воронка продаж с дублями — ломает все отчёты (LTV, конверсию).
- Сложная первичная настройка — админы не справляются, уходят клиенты.
- Много функционала, которым никто не пользуется (60+ отчётов).
- База знаний непригодна к использованию.
- ДДС технически слишком сложен для ввода расходов, Наташа вынуждена вести его отдельно в Excel.
- Главная причина оттока клиентов: "сложно вести, администраторы не разобрались, некогда".

## Принятые решения

1. **Сроки:** 4 месяца (февраль–май) на первую версию. В июне переезд текущих клиентов, июнь–август — обкатка и обратная связь, с августа — продажи новым клиентам.
2. **Ценообразование CRM:** модель "от абонементов" (как Альфа) или "от филиалов" (как Параплан) — решение отложено, нужен дополнительный анализ конкурентов.
3. **Миграция текущих клиентов:** уведомить за месяц, дать 2 недели на адаптацию, новых клиентов сразу заводить в новую систему.
4. **Ценообразование занятий:** ставка зависит от направления (не от педагога). В 95% случаев единая цена, редко — разная по направлениям.
5. **Скидки:** три типа — постоянная (фиксированная), разовая, связанная (действует пока выполняется условие, например второй ребёнок). Уведомления при отмене связанной скидки.
6. **Зарплата педагогов:** три схемы — сдельная (за ребёнка), часовая ставка, комбинированная (база + доплата за 4+ детей). Начисление помесячно, выплата — по любым датам клиента.
7. **Заморозки абонементов** — на текущем этапе игнорируем, мода ушла.
8. **Отчисление клиента:** остаток на балансе сохраняется, возврат не инициируется (кроме конфликтных ситуаций). Деньги можно перевести на другого ребёнка того же родителя.

## Состав MVP (первая версия)

**Входит:**
- Справочники (клиенты/родители, дети, педагоги, кабинеты, услуги/направления)
- Расписание и группы обучения
- Посещение, прогулы, перерасчёт
- Календарный абонемент
- Воронка продаж (лид → пробник → продажа)
- Баланс клиента, долги, переплаты
- ДДС (приход/расход, нал/безнал)
- Скидки (все три типа)
- Ввод зарплатных ставок педагогов (первичка)
- Простой ввод расходов для админов

**Не входит (отложено):**
- Пакетные абонементы (Левита) — после обкатки, ориентировочно август+
- Интеграция с платёжными системами (ЮKassa, Робокасса)
- SMS/мессенджер-уведомления
- Мобильная версия
- Онлайн-запись и личные кабинеты клиентов
- Склад/канцтовары
- Сложные отчёты (финрез, рентабельность по группам, ROMI)

## Открытые вопросы

1. **Модель ценообразования CRM** — от филиалов? от абонементов? комбинация? Нужен анализ Альфы, Параплана, 1С:Листок.
2. **Автоматические задачи для админов** — система должна формировать задачи по датам связи, оплаты, должникам. Детали реализации не обсуждались.
3. **Рассылки клиентам** — Telegram, Viber? Подрядчиков нет, WhatsApp API умер. Отложено минимум на полгода.
4. **Фильтры в отчётах** — сейчас отсутствуют, критично для работы.
5. **Как обрабатывать ситуации с задолженностью** — одно занятие в минус допустимо, дальше не пускаем. Нужно поле "обещанная дата оплаты".
