## 2. Ключевые находки по ролям

### 2.1. Оля — операционный менеджер

**Ежедневная рутина:**
- Проверяет неявки (раздел «Пропуски») — обрабатывает весь накопившийся список, не только за вчера
- Вносит платежи из Google Sheets ДДС (безнал — от Насти, наличные — от педагогов) в CRM
- Переводит новых учеников в «Актив» при наличии оплаты
- Заполняет данные из анкет (ФИО, контакты, ДР подопечных)
- Проводит отработки
- Переносит переплаты с выбывших абонементов на активные (раз в неделю)

**Ежемесячная рутина:**
- Создает расписание на следующий месяц
- Проверяет пустые группы (закрывает по согласованию с Настей)
- Выгружает зарплату в Excel для Насти → Наташа потом вносит обратно в CRM
- Вручную считает зарплату по уборке (ставка × количество дней в определенные дни недели)
- Вручную считает зарплату одного педагога с особой ставкой на конкретную группу

**Боли:**
- Слишком много последовательных действий для простых операций (скидка: создать → найти синюю надпись → нажать «Применить» → записать)
- Большой объем времени на внесение платежей в начале месяца (большая база)
- Переход на новое расписание (сентябрь) — занимает неделю+
- Невозможно сразу перевести выбывшего в «Лиды» с датой связи — сначала «Выбыл», потом вручную в «Лиды»
- Нельзя задать разную ставку педагогу на конкретную группу (только на направление)
- Прогулы за прошлый месяц не подтягиваются в расчет текущей зарплаты

**Что починила бы:**
- Объединить «Клиенты» и «Воронка продаж» обратно в один список. Пользователи путаются.

### 2.2. Наталья — финансовый менеджер

**Ключевой инсайт: роль не «бухгалтер», а финансовый менеджер.** Бухгалтерский контур (налоги) минимален и не в CRM. Основная работа — разнесение расходов, закрытие месяца, финрез.

**Три клиента — три разных процесса:**

| | «Умный я» (Анна) | Фирова | Горбунова |
|---|---|---|---|
| **Учет** | По кабинетам + направлениям | По филиалам (без кабинетов) | По филиалам + направлениям |
| **Доходы** | Вносит Оля | Наташа каждый вечер (1 час) | Помощница |
| **Расходы** | Раз в месяц из ДДС | Раз в месяц из ДДС | Раз в месяц + оперативные задачи |
| **Склад** | Приход → перемещение → списание по формам педагогов | Инвентаризация раз в месяц (админы считают фактически) | Нет учета по канцтоварам |
| **Зарплата** | 2 раза/мес | 2 раза/мес | 2 раза/мес + согласование с каждым педагогом |
| **Закрытие** | 1 день (к 10-му числу) | 1 день (к 15-му числу) | 1 день (к 10-му числу) |
| **Объем** | X | ~X + ежедневные платежи | 3X |

**Процесс закрытия месяца:**
1. Разнести все расходы из ДДС в CRM (филиал, кабинет, направление, месяц расхода, период амортизации)
2. Начислить налоги
3. Провести закрытие месяца (два этапа: акты выполненных услуг + стандартное закрытие УНФ)
4. Проверить финрез

**Боли:**
- **Нельзя провалиться из финреза в расшифровку.** «Через 33 колена проваливаться, смотреть, почему это так.» Использует отдельный отчет «Доходы и расходы по статьям (по отгрузке)» для проверки
- Много ручной работы по канцтоварам (приход, перемещение, списание)
- У Горбуновой: постоянные коллизии по зарплате из-за ошибок админов (пропущенные занятия, неверные группы)
- Невозможность автоматически начислить зарплату за пробные, когда вся группа — пробники (специфика Горбуновой)

**Что починила бы:**
- Drill-down из финреза: кликнул на цифру — увидел, из чего она складывается.

### 2.3. Настя Смирнова — старший админ / управляющий

**Ежедневная рутина:**
- Проверяет пробники (подтверждены ли на сегодня, обработаны ли вчерашние)
- Проверяет продажи (кто выходит на первое платное занятие)
- Смотрит пропуски (комментарии)
- Работа с должниками (первая половина месяца)
- Мониторит лиды (даты связи)
- Строит отчет «Воронка продаж» для педагогов (ежедневная статистика)
- Отмечает явки/неявки за одного педагога

**Ежемесячная рутина:**
- Детализация оттока → список для педагогов
- Свободные места → контент-план
- Зарплата менеджеров по каналам → расчет премии админов (вручную, пофамильно)
- Сравнение активных абонементов (сверка)
- Конверсия оттока по педагогам (пару раз в год с Аней)

**Боли:**
- **Комментарии в «работе с должниками» не очищаются при смене месяца.** Каждый месяц вручную заходит к каждому ученику и чистит комменты
- Карточка ученика «не очень удобная» (не может сформулировать конкретно)
- Частичная выгрузка базы (пропуск братьев/сестер)

**Что починила бы:**
- Кнопка «Очистить все комментарии в работе с должниками».

**Вне CRM:**
- Обзвоны: выгрузка базы → Google Sheets → обзвон с цветовой маркировкой → записавшиеся вносятся в CRM
- Задачи: Google Sheets (Настя → админы)
- ДДС: Google Sheets
- Канцтовары: Google Sheets

**Инсайт Анны (в момент интервью):**
- Автоматический перевод в «Актив» при оплате (без ручного клика)
- Обзвон должен быть в CRM без выгрузки → статусы меняются внутри, и только при конверсии лид «улетает куда надо»
- Сравнение «актив-актив» не нужен, если при оплате автоматом попадает в актив

### 2.4. Настя (админ)

**Ежедневная рутина:**
- Воронка: проверка пробников, лидов, продаж
- Запись лидов (через плюсик)
- Назначение пробных → SMS-визитка напоминает за сутки → подтверждение/перевод в продажу
- Пропуски: комментарии о причинах
- Расчет абонемента: расчет оплаты для клиентов
- Работа с должниками: по филиалам

**Боли:**
- **Две вкладки «Воронка продаж» и вторая** — путаница. Раньше было одно, потом разделили. Забывает переключить, видит неполный список.

**Что починила бы:**
- Убрать разделение вкладок «Воронка продаж» / «Клиенты». Сделать единый список (как было).

**Вне CRM:**
- Обзвоны из Google Sheets (получает от Насти, обзванивает, помечает цветами)
- Задачи в Google Sheets
- Шаблоны сообщений/рассылок в Google Sheets

### 2.5. Виктория — админ (центр Горбуновой)

**Ежедневная рутина:**
- Обработка лидов: звонок → заметки на бумажке → обработка в голове → внесение в CRM
- Пробники: назначение, подтверждение, перевод в продажу/лиды/потенциал/предзапись
- Расчет абонемента для новых — **вручную на календаре** (пока ученик не в активе, CRM ничего не считает)
- Рассылка «писем счастья» (оплаты) — использует 3 вкладки: клиенты + оплаты + работа с должниками
- Ежевечерний отчет руководителю: актив, активные абонементы, непродленные, разница, отток
- Инвентаризация (редко)

**Особенности:**
- Знает расписание наизусть, редко заглядывает в CRM для этого
- Оплаты в CRM заносит не она, а бухгалтер (из их файла)
- Активно пользуется отчетами: активные/непродленные абонементы, сравнение, детализация оттока, конверсия оттока по педагогам, отток по направлениям, доходимость, свободные места, оплаты

**Боли:**
- **Недостаточная «кликабельность» между отчетами.** Из детализации оттока нельзя перейти в корректировку — нужно открывать отдельно.
- При удалении ученика не проставляется фамилия ответственного автоматически
- Назначение ответственного за пробник — лишние шаги через «Занятия»
- Расчет первого абонемента до перевода в актив — вручную

**Что починила бы:**
- Drill-down / навигация между отчетами.

**Вне CRM:**
- Задачник (CRM-задачи неудобные)
- ДДС
- Отзывы
- Канцтовары

### 3.2. Паттерны «все хорошо»

- **Никто не может вспомнить серьезный сбой системы.** Данные не теряются, расчеты корректные.
- **Все привыкли.** Даже неудобные процессы воспринимаются как норма.
- **Отчеты — сильная сторона.** Виктория активно использует 10+ отчетов. Все пользуются воронкой, оттоком, должниками.

### 4.2. Рекомендуемые изменения в PRD

#### Высокий приоритет (влияет на MVP)

**1. Единый список клиентов/лидов вместо двух вкладок**
- **Текущий PRD:** Воронка лидов (CRM-04) + Работа с клиентской базой (CRM-05) — описаны как два раздела
- **CustDev:** 3 из 5 респондентов (Оля, Настя С., Настя А.) назвали разделение вкладок главной проблемой. Пользователи переключаются на «Все» и работают из единого списка.
- **Рекомендация:** Один экран с фильтрами по статусу (лид / актив / выбывший / ЧС / архив). Воронка продаж и работа с клиентской базой — визуальные фильтры/вкладки внутри одного экрана, а не отдельные разделы. По умолчанию показывать «Все».

**2. Drill-down во всех отчетах**
- **Текущий PRD:** Не описан как требование
- **CustDev:** Наташа (#1 боль): «через 33 колена провалиться». Виктория: «кликабельность между отчетами»
- **Рекомендация:** Добавить требование: **все суммы в отчетах должны быть кликабельными** — клик раскрывает расшифровку до первичных документов. Отдельно: из финреза → в расходы/доходы по статьям → в конкретные операции.
- **Предлагаемый ID:** FIN-26 (MVP)

**3. ДДС внутри CRM (а не в Google Sheets)**
- **Текущий PRD:** Приём оплат (FIN-01), расходы (FIN-02) уже описаны. Но не акцентировано, что ДДС должна быть единственным местом учета денег
- **CustDev:** Сейчас педагоги/админы вносят в Google Sheets → Оля/Наташа переносят в CRM. Двойная работа.
- **Рекомендация:** Описать принцип «единый контур ДДС»: наличные и безнал вносятся только в CRM. Педагоги и админы вносят платежи напрямую. Не нужен отдельный Google Sheet.
- **Связь:** Это не новое требование, а усиление FIN-01 + FIN-08. Нужно явно прописать, что ДДС — это не отчет, а центральный процесс.

**4. Обзвоны / кампании — базовая версия**
- **Текущий PRD:** Не описан (ближайшее — CRM-18, сортировка лидов + автозадачи CRM-08)
- **CustDev:** 3 из 5 респондентов работают с обзвонами в Google Sheets. Процесс: выгрузка → таблица → обзвон → цветная маркировка → возврат в CRM.
- **Рекомендация:** Добавить требование «Списки обзвона» — возможность создать список лидов/клиентов по фильтрам (возраст, филиал, статус, дата выбытия), обзвонить по списку с пометками статусов (цвет / комментарий / дата перезвона) **без выгрузки из системы.**
- **Предлагаемый ID:** CRM-20 (v1.1 или MVP — обсудить с Анной)

**5. Должники: обещанная дата оплаты как отдельное поле, а не комментарий**
- **Текущий PRD:** Уже есть «Обещанная дата оплаты» (US-021)
- **CustDev:** Настя С. каждый месяц вручную чистит комментарии должников. Потому что «обещанная дата» живет в комментариях.
- **Рекомендация:** Акцентировать в PRD: обещанная дата — это **структурированное поле**, а не текстовый комментарий. При смене месяца: автоматический сброс просроченных обещаний (или уведомление). Комментарии по должникам — отдельно от общих комментариев клиента.

**6. Ставка ЗП на конкретную группу (не только направление + филиал)**
- **Текущий PRD:** SAL-04 — привязка к сотрудник + филиал + направление
- **CustDev:** Оля вручную считает зарплату педагогу с особой ставкой на одну группу. Виктория подтверждает: «видела в других CRM — ставка на конкретную группу»
- **Рекомендация:** Расширить SAL-04: привязка ставки к сотрудник + филиал + направление + **группа (опционально)**. Если указана группа — она приоритетнее, чем общая ставка на направление.

#### Средний приоритет (v1.1 или уточнение)

**7. Задачи: сделать удобными**
- **Текущий PRD:** DSH-02, DSH-03 — задачи ручные + автоматические
- **CustDev:** Все используют Google Sheets для задач, потому что встроенные задачи «неудобные»
- **Рекомендация:** При проектировании задач учесть: простота (создать за 2 клика), фильтр «мои на сегодня», привязка к клиенту/ученику, делегирование (Настя → админ). Это не новое требование, но важен UX-фокус.

**9. Расчет первого абонемента до перевода в актив**

**10. Прогулы за прошлый месяц в расчете текущей зарплаты**
- **Текущий PRD:** Не описан
- **CustDev:** Оля: «за прошлый месяц не подтягиваются прогулы, и я каждый раз иду в предыдущий месяц, смотрю переплаты/недоплаты педагогам»
