# Логика расчётов отчётов

> **Источник:** Отчёты.csv (описание ~60 отчётов из текущей 1С) + экспорт базы знаний Tilda
> **Дата:** 15.03.2026, обновлено 17.03.2026 (ревью Анны)
> **Назначение:** Справочник формул и логики для реализации. На каждой странице отчёта — кнопка «Справка» с описанием расчёта

---

## 1. CRM и клиенты

### 1.1. Активные абонементы

**Что показывает:** Количество абонементов, по которым в текущем месяце было хотя бы 1 занятие со списанием.

**Столбцы:**
- **Количество абонементов за месяц** — абонементы с хотя бы 1 списанием в текущем месяце. На 1 число = 0, растёт по мере отмечания посещений
- **Продлённые абонементы** — абонементы, у которых было хотя бы 1 списание в прошлом месяце И хотя бы 1 списание в текущем. Отработки прошлого месяца НЕ влияют — только занятия по расписанию (новый абонемент)
- **Активные на конец месяца** — абонементы с хотя бы 1 списанием в текущем месяце, где клиент остался активным на последнее число. Если сегодня 15-е — смотрим активность на сегодня

**Детализация:** до филиала → направления → клиента (двойной клик → «лид»)

**Баг в 1С:** Отработанные занятия прошлого месяца влияют на столбец «продлённые». Не должны.

---

### 1.2. Непродлённые абонементы

**Что показывает:** Клиенты, активные в прошлом месяце (хотя бы 1 списание), но без списаний в текущем.

**Логика:** На 1 число — все активные прошлого месяца. По мере отмечания посещений клиенты исчезают из отчёта.

**Рекомендация:** Пользоваться после 8 числа месяца.

**Поле комментария:** Администратор может добавить комментарий по возвращению клиента.

---

### 1.3. Средний чек

**Уровень:** ДДС (движение денежных средств). НЕ связан с абонементами.

**Формула:**
```
Средний чек = Сумма всех платежей / Количество платежей
```

**Определения:**
- **Сумма всех платежей** — сумма всех поступлений от клиентов за выбранный период
- **Количество платежей** — количество всех платежей клиентов. Один платёж = один чек, даже если он покрывает несколько абонементов

**Пример:** Родитель платит 10 000₽ за двоих детей → это один платёж (чек) на 10 000₽ → при этом два абонемента (6 000₽ + 4 000₽). Средний чек = 10 000₽.

**Нюансы:**
- Если клиент платит за 2 направления/детей одним платежом — это 1 платёж (увеличивает средний чек)
- Если клиент платит абонемент частями (3000 = 3×1000) — это 3 платежа (уменьшает средний чек)

---

### 1.4. Средняя стоимость абонемента

**Формула:**
```
Средняя стоимость абонемента = Сумма отработанных / Количество активных абонементов за месяц
```

**Определения:**
- **Количество активных абонементов за месяц** — абонементы, по которым было хотя бы 1 списание (charge) в течение месяца. Не на начало и не на конец — любой, у кого было хотя бы 1 списание в этом месяце, включая отчисленных
- **Сумма отработанных** — сумма фактически отработанных (списанных) сумм по всем активным абонементам

**Пример:** Ребёнок оплатил абонемент 4 000₽, ушёл в середине месяца, отработано 2 000₽ → стоимость этого абонемента для расчёта = 2 000₽ (не 4 000₽).

**Нюансы:**
- Клиенты, пришедшие в середине месяца, снижают средний абонемент (меньше отработанных занятий)
- Отчисленные до конца месяца — считается только фактически отработанная сумма

---

### 1.5. Сегментация клиентов

**Правило:** Автоматически по количеству купленных абонементов.

| Сегмент | Абонементов |
|---|---|
| Новый | 1–3 |
| Стандарт | 4–12 |
| Постоянный | 13–18 |
| VIP | 19+ |

---

### 1.6. LTV клиентов в разрезах

**Измеряется в двух единицах:** деньги (сумма поступлений) И количество абонементов (= месяцев).

**Столбцы:**
- **Количество занятий со списаниями** — все занятия с любым списанием (явки, прогулы, частичное). НЕ учитываются полные перерасчёты и неявки
- **Длительность посещений в месяцах** — количество месяцев, в которых было хотя бы 1 списание. Месяцы без списаний не считаются
- **Количество посещённых направлений** — уникальные направления с хотя бы 1 списанием. Повторный приход на то же направление = 1 направление
- **Сумма всех поступлений** — все платежи клиента (без учёта скидок и корректировок)
- **Сумма занятий с фактическим списанием** — только сумма списаний по отмеченным занятиям

---

### 1.7. Сверка актива (расхождения)

**Назначение:** показать разницу между клиентами «в активе» и реальной активностью (оплата/посещения). Используется администратором с 1 по 10 число для выявления «мёртвых душ».

**Ключевой термин:** **Активированный абонемент** — абонемент, с которого начались списания (есть хотя бы одно посещение). Отличается от просто оплаченного.

**Столбцы:**
| Столбец | Формула / источник |
|---|---|
| Клиент | Client.name |
| Подопечный | Ward.name |
| Направление | Direction.name |
| Группа | Group.name |
| В активе | Client.status = active |
| Абонемент оплачен | Есть Payment за текущий месяц для данного абонемента |
| Абонемент активирован | Есть хотя бы одно Attendance с charged_amount > 0 за текущий месяц |
| Последнее посещение | MAX(Attendance.date) WHERE status = present |
| Дней без посещений | TODAY − Последнее посещение |
| Расхождение | В активе = да, но оплата = нет И активирован = нет |

**Фильтры:** период, филиал, направление, только расхождения (переключатель).

**Сортировка по умолчанию:** по количеству дней без посещений (убывание).

**Бизнес-правило:** отчисление всегда вручную администратором. Система только показывает расхождения, не отчисляет автоматически.

## 2. Работа с оттоком

### 2.1. Детализация оттока

**Что показывает:** Все выбывшие абонементы за выбранный период. Выбывший = клиент, не купивший следующий абонемент (по факту прекращения оплат). Административное отчисление — отдельное действие, в отчётах оттока не используется.

**Логика:** Дата оттока = дата ПОСЛЕДНЕГО ПЛАТНОГО занятия, НЕ дата уведомления клиента об уходе и НЕ дата административного отчисления.

**Пример:** Последнее платное занятие 25 мая, клиент сообщил об уходе 5 июня → отток относится к маю, не к июню.

**Управление:** Галочки «отток по направлению» и «отток по педагогу» при выбытии управляют попаданием в соответствующие отчёты.

**Типы оттока:**
- «Ушёл с направления» — не закончил курс
- «Закончил курс обучения» — прошёл все занятия программы

---

### 2.2. Конверсия оттока по педагогам

**Формула:**
```
% оттока = Отток / Активные абонементы
```

**Определения:**
- **Активные** — количество активных абонементов у педагога
- **Отток** — количество выбывших (не купили следующий абонемент)

**Нюанс:** Если в середине месяца у ученика сменился педагог, активные считаются обоим → итоговая сумма может быть больше оттока по направлениям.

**Второй вид:** Через переключатель — отток в разрезе филиалов.

---

### 2.3. Отток по месяцам

**Что показывает:** В какой месяц «срока жизни» чаще всего уходят клиенты.

**Формула:**
```
Месяц оттока = (Дата последнего платного занятия − Дата продажи) в полных календарных месяцах
```

**Дата продажи** = дата внесения денег ИЛИ дата первого платного занятия (что наступило раньше).

**Примеры:**
- Продажа 01.05, последнее платное 25.05 → 1-й месяц
- Продажа 01.05, последнее платное 02.06 → 2-й месяц

**Баг в 1С:** Если не стоит дата продажи, клиент в отчёт не попадает.

---

### 2.4. Отток по направлениям (и филиалам)

**Столбцы:**
- **Активные абонементы** — хотя бы 1 списание в прошлом И текущем месяце
- **Отток** — выбывшие (не купил следующий абонемент + закончил курс)
- **% оттока** = Отток / Активные
- **Закончили курс** — отдельный столбец (только «окончил курс обучения»)

**Вариант «по филиалам»:** То же самое + деление по филиалам.

## 3. Работа с лидами

### 3.1. Детализация пробников

**Что показывает:** Все абонементы с назначенным пробным занятием.

**Нюансы:**
- Серым подсвечиваются повторные пробные по тому же абонементу
- Разовые клиенты не учитываются

---

### 3.2. Не пришли на пробники

**Что показывает:** Клиенты, не посетившие назначенное пробное (статус «Не пришёл»). Каждая неявка считается отдельно. Отменённые записи (технический статус при переносе даты или удалении заявки) не учитываются.

---

### 3.3. Конверсия пробных по педагогам

**Формула:**
```
Конверсия = Количество продаж / Количество пробных
```

**Определения:**
- **Количество пробных** — клиенты, посетившие пробные у педагога
- **Количество продаж** — купившие после пробного. Купивший = первое платное занятие посещено ИЛИ оплата внесена (что наступило раньше). Сам факт покупки абонемента без посещения и без оплаты НЕ считается конверсией

**Баг в 1С:** Продажи считаются по галочке «продажа совершена», а не по первому платному занятию.

---

### 3.4. Лиды по дням

**Что показывает:** Количество созданных абонементов в каждый день месяца в разрезе каналов привлечения.

**Фильтры:** филиал, направление.

---

### 3.5. Пробники по дням

**Столбцы (по дням):**
- **Записались на пробное** — дата, когда сотрудник назначил пробное (не дата самого пробного!)
- **Посетили пробное** — дата фактического посещения
- **Купили абонемент** — дата проставления галочки «продажа совершена»

**Всё в разрезе каналов привлечения.**

**Баг в 1С:** «Купили абонемент» не имеет ничего общего с первым платным занятием — привязано к дате продажи.

---

### 3.6. Лиды по менеджерам

**Столбцы (по дням, в разрезе менеджеров):**
- **Создано лидов** — по дате создания контакта
- **Создано заявок** — по дате создания абонемента
- **Записано на пробники (лидов)** — дата действия сотрудника (не дата пробного!)
- **Записано на пробники (заявок)** — аналогично по абонементам
- **Совершено продаж (лидам)** — по дате продажи
- **Совершено продаж (абонементам)** — по дате продажи
- **Отток клиентов** — по дате отчисления (последнее платное или вручную)
- **Отток абонементов** — аналогично

**Баг в 1С:** «Создано лидов» показывает не создание контакта, а контакта с направлением.

---

### 3.7. Доходимость (по дням)

**Модель «по дате события»** (не когортная!): каждая строка считается по дате своего события, а не по дате создания заявки. Отчёт на **двух вкладках**, заявки разделены полной партицией по наличию пробного:
- заявка имеет ≥1 пробное (`TrialLesson.applicationId`) → вкладка **«С пробным»**;
- заявка без пробного → вкладка **«Без пробного»**.

**Конверсия — каскадная (этап к предыдущему).**

**Вкладка «С пробным»** (полная воронка):
| Строка | По какой дате | Конверсия |
|---|---|---|
| Создано заявок | `Application.createdAt`, только заявки **с** пробным | — |
| Записано на пробник | `TrialLesson.scheduledDate` | записано / создано |
| Посетил пробник | `TrialLesson.attendedAt` | посетил / записано |
| Совершено продаж | дата продажи (см. ниже) | продаж / посетил |

**Вкладка «Без пробного»** (короткий цикл):
| Строка | По какой дате | Конверсия |
|---|---|---|
| Создано заявок | `Application.createdAt`, только заявки **без** пробного | — |
| Совершено продаж | дата продажи (см. ниже) | продаж / создано |

**Продажа** (обе вкладки):
- первый абонемент клиента (`previousSubscriptionId = null`, т.е. не продление);
- **дата продажи = самая ранняя из**: дата зачисления денег на абонемент (`Payment` типа `incoming` ИЛИ `transfer_in` — оплата с баланса родителя проводится как `transfer_in` и это основной способ пополнения) и дата первого платного занятия (`Attendance`: `isTrial=false`, `isPending=false`, `chargeAmount>0` → `Lesson.date`);
- абонемент **без** оплаты и **без** платного занятия в отчёт **не попадает**;
- разделение по вкладкам: было ли у подопечного пробное **в направлении абонемента** (по `wardId`, либо по `clientId`, если у абонемента нет подопечного). Направление пробного берётся из `TrialLesson.directionId`, а если он пуст — из направления группы пробного (`group.directionId`): у большинства пробных своё поле направления не заполнено.

**Замечания:**
- Связи `Subscription`/`Payment` → `Application` в БД нет, поэтому продажа относится к вкладке по истории подопечного, а не по конкретной заявке.
- На вкладке «С пробным» конверсия «Записано/Создано» близка к 100% (по определению все заявки этой вкладки имеют пробное), но может отличаться: пробные считаются по своей дате в периоде, заявки — по дате создания.
- Строки «Поступила оплата» нет (убрана: дублировала продажу).

**Дополнительная детализация:**
- Количество пробных по каждому каналу привлечения
- Кто пришёл на пробное, кто не пришёл
- Кто оплатил после пробного
- Разбивка по дням

---

### 3.8. Доходимость (свод)

**Детализация до ФИО.** Столбцы:
- Лид (ФИО), заявка (направление), дата создания, состояние лида
- Дата пробника
- Создано заявок (да/нет)
- Записано на пробник (да/нет)
- Посетил пробник (да/нет)
- Совершено продаж (да/нет)
- Поступила оплата (да/нет)

**Баг в 1С:** Если клиент заведён в 2023 и переведён в актив в 2025 — в отчёте он будет в 2023. Оплата отображается даже при неполной оплате.

### 3.9. Воронка продаж (CRM-13)

**Событийная воронка по заявкам за месяц.** Реализация: `app/src/lib/reports/sales-funnel.ts`, страница `/reports/crm/funnel`.

**Две вкладки** (по статусу клиента НА МОМЕНТ создания заявки):
- **Новые** — клиент был Лидом/Потенциалом (не покупал до создания заявки)
- **Действующие** — клиент был Активным/Выбывшим (уже покупал): допродажи и возвраты

Признак «покупал до даты X»: min(firstPaymentDate, firstPaidLessonDate, processedAt выигранной заявки, дата активации абонемента) < X. Фолбэк для импортированных без истории — текущий clientStatus (active/churned).

**Две схемы в каждой вкладке** (по наличию у заявки не-отменённого пробного):
- С пробным: Лид → Заявка → Пробное → Пришёл на пробное → Купил
- Без пробного: Лид → Заявка → Купил
- (на вкладке «Действующие» этапа «Лид» нет)

**Даты этапов:**
- Лид — контакты, **СТАВШИЕ лидом** (вошедшие в статус «Новый») в выбранном месяце: `Client.becameLeadAt` в месяце, каждый один раз. Считается момент входа в статус, а НЕ текущий статус — лид, конвертированный за месяц в актив/потенциал/др., остаётся в счёте своего месяца. `becameLeadAt` ведёт триггер БД `set_client_became_lead_at` (при создании со статусом new → created_at; при возврате в new → now()). Залитые импортом сразу выбывшими/потенциалом/архивом becameLeadAt не имеют и не считаются. Лид ещё без заявки — цифра одинакова в обеих схемах
- Заявка — `Application.createdAt` (deletedAt = null)
- Пробное / Пришёл на пробное — `TrialLesson.scheduledDate` (дата занятия); заявка считается один раз за месяц
- Купил — заявка выиграна (`processedToStatus=won`, дата `processedAt` = оплата абонемента) ЛИБО первое платное занятие (`Client.firstPaidLessonDate`, относится к самой ранней заявке клиента без won)

**Два разреза каждого этапа:**
- Текущий месяц — заявка создана в выбранном месяце
- Перетекающие — заявка создана раньше, действие совершено в этом месяце
- У «Лид» и «Заявка» перетекающих нет (их действие = создание)

**Детализация:** клик по этапу — список заявок (родитель, телефон, ребёнок, филиал, направление, группа; для «Лид» — ребёнок/родитель/телефон). Перетекающие помечены.

**Дашборд:** виджет «Воронка продаж» показывает те же этапы одной суммарной цифрой (текущие + перетекающие).

### 3.10. Эффективность обзвонов

**Назначение:** оценка результативности обзвонных кампаний.

**Столбцы:**
| Столбец | Формула / источник |
|---|---|
| Кампания | CallCampaign.name |
| Период | CallCampaign.created_at — CallCampaign.closed_at |
| Всего в списке | COUNT(CallCampaignItem) |
| Отработано | COUNT(CallCampaignItem WHERE status != pending) |
| Назначено пробных | COUNT(CallCampaignItem WHERE result = trial_scheduled) |
| Продажи | COUNT(CallCampaignItem WHERE result = sale) |
| Не дозвонились | COUNT(CallCampaignItem WHERE result = no_answer) |
| Отказы | COUNT(CallCampaignItem WHERE result = refused) |
| % отработки | Отработано / Всего × 100% |
| Конверсия в пробное | Назначено пробных / Отработано × 100% |
| Конверсия в продажу | Продажи / Отработано × 100% |

**Фильтры:** период, администратор, кампания.
**Группировка:** по кампании, по администратору.

---

### 3.11. Продажи менеджеров по каналам

**Три показателя (переключаемые):**
- **Назначенные пробники** — по дню назначения
- **Посещённые пробники** — по дате посещения
- **Продажи абонементов** — по дате продажи

В разрезе сотрудников и каналов привлечения.

**Баг в 1С:** Данные подтягиваются даже если клиент не переведён в актив и не внёс платёж.

## 4. Загруженность

### 4.1. Свободные места в группах

**Столбцы:**
- **Всего мест** — из настроек группы (макс. кол-во)
- **Занято мест** — закреплённые активные ученики на сегодня
- **Записано на пробники** — лиды с назначенным пробным, не в потенциале
- **Ждём оплату** — после пробного сказали, что будут ходить, но ещё не внесли деньги и не пришли на первое платное занятие

**Формулы:**
```
Свободно мест = Всего мест − Занято мест − Записано на пробники − Ждём оплату
% заполнения = (Всего мест − Свободно мест) / Всего мест
```

**Пример:** В группе 8 мест, занято 4, на пробнике 1, ждём оплату 2 → свободно 1. Если ожидающие не оплатят — свободных станет 3.

---

### 4.2. Загруженность центра

**Столбцы:**
- **Максимальное кол-во часов** — из настроек филиалов/кабинетов (часы работы × рабочие дни)
- **Фактическое кол-во часов** — сумма длительности занятий с учениками из расписания

**Формула:**
```
% загрузки = Фактическое кол-во часов / Максимальное кол-во часов
```

**Нюанс:** Пустые занятия (без учеников) НЕ учитываются в фактических часах.

---

### 4.3. Лист ожидания

**Что показывает:** Клиенты, ожидающие место (направление, педагога, время, группу).

**Автоматика:** При отчислении ученика из группы → уведомление, что место ожидает другой клиент (зелёная подсветка «есть свободные места»).

## 5. Деньги

### 5.1. Оплаты

**Столбцы:**
- **Баланс на начало периода** — состояние счёта абонемента на 1 число
- **Количество занятий** — из расписания за месяц
- **Сумма абонемента** — кол-во занятий × стоимость каждого
- **Оплачено** — сумма платежей в выбранном месяце
- **Баланс на конец периода** — состояние на последнее число
- **Дата оплаты** — дата последней оплаты в месяце
- **Нал/безнал** — способ оплаты
- **Сумма без скидок** — сумма занятий без учёта скидок
- **Сумма скидки** — все скидки за месяц (разовые + постоянные)
- **Наши деньги** — сумма занятий со списанием (по мере отмечания)
- **Педагог** — педагог группы на последнее число
- **Группа обучения** — группа на последнее число

**Баг в 1С:** «Оплачено» показывает не оплату за конкретный месяц, а оплату, произведённую В месяце (клиент мог заплатить за 2 месяца сразу).

---

### 5.2. Расчёты с учениками

**Столбцы:**
- **Начальный баланс** — на 1 число
- **Начисление План** — сумма за весь месяц (занятия × стоимость, перерасчёты вычтены)
- **Начисление Факт** — сумма отмеченных занятий со списанием
- **Оплата** — сумма платежей в месяце
- **Конечный баланс** — на последнее число

---

### 5.3. Ожидаемые поступления

**Столбцы:**
- **Сумма абонементов** — сколько все клиенты должны за месяц по расписанию
- **Ожидаемые поступления** — неоплаченные абонементы (частично и полностью) за текущий и предыдущий месяц, только по активным. Долги отчисленных НЕ попадают
- **Оплачено** = Сумма абонементов − Ожидаемые поступления
- **% долга** = Ожидаемые поступления / Сумма абонементов
- **Сумма скидок** — итоговая разница между базовой ценой и ценой со скидкой за месяц

**Баг в 1С:** Нет возможности закрепить оплаты за конкретным месяцем.

**Прогноз на следующий месяц (решение 19.03.2026):**
- Прогноз из выписанных абонементов активной базе на следующий месяц
- Меняется ежедневно (новые клиенты, отчисления)
- Формула: сумма стоимостей активных абонементов на текущую дату

---

### 5.4. Календарь постоянных платежей

**Столбцы:**
- **Статья ДДС** — статья расхода
- **Аналитика** — ФИО сотрудника (для ЗП)
- **План платежей** — плановая сумма (заполняется вручную)
- **Оплачено** — факт оплаты в текущем месяце
- **К оплате** = План − Оплачено

**Нюанс:** ЗП педагогов вносить не нужно — подтягивается автоматически из ставок.

**Баг в 1С:** Если аренду оплатили в мае за июнь, сумма попадёт в май (по дате платежа, не по периоду).

### 5.5. Движения денежных средств (ДДС) — журнал

**Что показывает:** Построчный журнал ВСЕХ денежных операций за период (как «Лайф Планер»). НЕ свод по категориям — каждая транзакция отдельной строкой.

**Принцип:** По дате фактического платежа, НЕ по периоду назначения.
- Клиент оплатил 18.06 за июль → в ДДС попадёт в июнь.
- Аренда за сентябрь оплачена 18.06 → в ДДС попадёт в июнь.

**Источники строк журнала** (UI: `/finance/dds`):
1. `Payment` — приход (`incoming`) / возврат (`refund`). Контрагент = клиент или «Прочий доход» (для платежей без client_id).
2. `Expense` — расход (отрицательная сумма). Категория = `ExpenseCategory.name`.
3. `SalaryPaymentItem` — каждая позиция документа выплаты ЗП = отдельная строка журнала (счёт × сумма × сотрудник). Direction в журнале не показывается.
4. `AccountOperation`:
   - `transfer` — перемещение между счетами, одна строка «Откуда → Куда»
   - `owner_withdrawal` — выемка собственника, отрицательная сумма
   - `encashment` — инкассация, перемещение касса → р/с

**Колонки журнала:** Дата, Тип (бейдж), Категория, Контрагент, Счёт (или «Откуда → Куда»), Сумма (+/−), Ответственный (`createdBy` → ФИО), Комментарий.

**Шапка страницы:** карточки счетов с балансом и ±/период (клик = фильтр журнала по счёту). Бейджи-фильтры по типу операции. Сводки: Поступления / Выбытия / Остаток на счетах.

**Бегущий остаток в строках НЕ показывается** — общие остатки видны в карточках счетов.

### 5.6. Остаток денег

**Что показывает:** Состояние всех счетов и касс на выбранный день.

---

### 5.7. Поступления денег по дням

**Что показывает:** Ежедневные поступления в разрезе нал/безнал.

**Фильтры:** только от клиентов / все платежи, по филиалу.

**Второй вид:** В разрезе конкретных счетов/касс.

### 5.8. Работа с должниками

**Столбцы:**
- **Дата** — по умолчанию последнее число месяца
- **Контрагент** — ученик
- **Договор** — направление
- **Состояние лида** — текущий статус
- **Сегмент лида** — сегментация (если настроена)
- **Есть актив** — есть ли активные абонементы (для переноса долга/переплаты)
- **Плановый долг** — сколько должен, если посетит все занятия (перерасчёты вычтены)
- **Фактический долг** — долг на сегодня по отмеченным занятиям
- **Источники долга** (Ф2/Ф2.8) — разбивка долга по абонементам/направлениям из `ClientBalanceTransaction`. Группировка по `subscriptionId` (если транзакция к нему привязана) или `directionId`. Помогает ответить на вопрос «по чему именно должен» — например, «3000₽ по направлению Танцы, абонемент за май»
- **Комментарий по оплате** — вручную
- **Группа обучения**

**Источник «фактического долга» после Ф2:** `Client.clientBalance` (если < 0, это и есть долг). Хранится с обратным знаком: −5000₽ = клиент должен 5000₽. История формирования долга — в `ClientBalanceTransaction` (типы `subscription_issued`, `personal_lesson_charge`, `refund`, `payment_received` и т.д.).

**Баг в 1С:** Если клиент оплатил май 5 июня → в должниках на 31.05 он будет в долгах. В новой системе оплата создаёт транзакцию `payment_received` с датой платежа, отчёт на 31.05 учитывает только транзакции до этой даты — баг устранён.

### 5.9. Остатки оплаченных занятий

**Что показывает:** Сколько занятий осталось по абонементу, контроль дат окончания.

**Формулы после Ф2** (`balance` хранится в ₽, не в занятиях):
- `Отработано занятий = round(subscription.chargedAmount / subscription.lessonPrice)` или эквивалентно `count(Attendance where chargeAmount > 0)`
- `Остаток занятий = subscription.totalLessons − Отработано` или `round(subscription.balance / subscription.lessonPrice)`

Деление целочисленное в норме (одно занятие = `lessonPrice`). При `chargePercent < 100` в матрице — Subscription всё равно списывает полную цену, разница возвращается на `clientBalance` (см. Ф2 — частичные списания).

**Нюанс:** У выбывших с остатком денег — остатки = 0 (при возврате стоимость может измениться). Ориентироваться по «Баланс на сегодня».

---

### 5.10. Отработанные абонементы

**Столбцы:**
- **Сумма абонемента** — кол-во занятий в расписании × стоимость (сколько должны за месяц). Данные реального времени: уменьшается при отчислении/перерасчёте, увеличивается при добавлении
- **Отработано** — сумма отмеченных занятий со списанием → попадает в финрез как доход

**Поле `subscription.chargedAmount`** (Ф2) — накопительная сумма всех списаний по абонементу в ₽. Растёт при каждой отметке `present`/`no_show`/`absent`, уменьшается при сбросе отметки или переносе занятия. Используется в этом отчёте напрямую.

**По итогу месяца:** При всех отмеченных посещениях суммы должны совпасть.

**Нюанс:** Перерасчёт уменьшает сумму абонементов.

---

### 5.11. Связанные скидки

**Столбцы:**
- Ученик (основание скидки) — из-за кого скидка
- Ученик (связанный) — у кого скидка
- Договор (основание) / Договор (связанный)
- Состояние лида (основание) / (связанного)
- Неактивен? — флаг неактивности основания/связанного
- Скидка — название
- Начало действия

## 6. Зарплата

### 6.1. Зарплата педагогов

**Принцип:** Зарплата состоит из автоматически рассчитанной части (по занятиям) + ручные премия/штраф в отдельных столбцах. Автоматический расчёт — это база, премия и штраф — дополнения, отображаемые отдельно.

**Столбцы:**
- **Сотрудник**
- **Начислено за месяц** — автоматический расчёт по отмеченным занятиям (растёт в реальном времени)
- **Начислено (аванс)** — автоматический расчёт за первую половину
- **Премия (аванс)** — ручная, отдельный столбец
- **Штраф (аванс)** — ручной, отдельный столбец
- **Выплачено (аванс)**
- **Начислено (зарплата)** — автоматический расчёт за вторую половину
- **Премия (зарплата)**, **Штраф (зарплата)**, **Выплачено (зарплата)**
- **Осталось выплатить** = Начислено + Премии − Штрафы − Выплачено

**Детализация:** клик → по дню → по направлению → по филиалу → до каждого занятия с временем.

**Неотмеченные занятия не учитываются.**

---

### 6.2. Прогноз сдельной оплаты

**Столбцы:**
- **Педагог** (только сдельные)
- **Филиал**, **Направление**
- **База** — условия начисления
- **Ставка** — стоимость по условиям
- **Количество (база)** — количество учеников на занятиях
- **Сумма прогноз** = Ставка × Количество (база)
- **Выплачено** — уже проведённые выплаты
- **К оплате** = Прогноз − Выплачено

**Данные реального времени:** При отчислении/продаже цифры обновляются.

---

### 6.3. Часы педагогов по дням

**Что показывает:** Отработанные часы за каждый день.

**Логика:** Если отмечен хотя бы 1 ученик с фактическим посещением → педагог отработал час. Данные в часах (30 мин = 0.5 часа).

**Баг в 1С:** Прогул считается как отработанное занятие, даже если ученик в группе один.

---

### 6.4. Средняя ЗП педагогов

**Формула:**
```
Средняя стоимость часа = Зарплата / Часы
```

**Столбцы:** Педагог, Часы (по отмеченным), Зарплата (начисленная), Средняя стоимость часа.

**Можно выбрать несколько месяцев для сравнения динамики.**

---

### 6.5. Мотивация администратора

**Назначение:** расчёт бонусов администраторов за продажи.

**Столбцы:**
| Столбец | Формула / источник |
|---|---|
| Администратор | Employee.name |
| Филиал | Branch.name |
| Пробных проведено | COUNT(TrialLesson WHERE status = completed AND created_by = admin) |
| Бонус за пробные | Пробных × AdminBonusSettings.amount (per_trial) |
| Продаж новым | COUNT(первых Subscription WHERE created_by = admin) |
| Бонус за продажи | Продаж × AdminBonusSettings.amount (per_sale) |
| Допродаж | COUNT(не первых Subscription WHERE created_by = admin) |
| Бонус за допродажи | Допродаж × AdminBonusSettings.amount (per_upsale) |
| Итого бонус | Сумма всех бонусов |
| Оклад | SalaryRate.amount (fixed) |
| Итого ЗП | Оклад + Итого бонус |

**Фильтры:** период, филиал, администратор.

## 7. Компания

### 7.1. Прогноз прибыли

**Формула:**
```
Прогноз прибыли = Сумма абонементов − Прогноз ЗП педагогов − Прогноз переменных расходов − Прогноз постоянных платежей
```

**Определения:**
- **Сумма абонементов** = все занятия учеников × стоимость каждого (реальное время)
- **Прогноз ЗП** = автомат из расписания + ставок
- **Прогноз переменных расходов** = средние расходы по канцтоварам/хозтоварам за предыдущие месяцы (со склада)
- **Прогноз постоянных** = заполняется вручную раз в месяц (план платежей)

---

### 7.2. Финансовый результат (P&L / ОПИУ)

**Структура:** Филиал → Направление → Статья расхода.

**Формулы:**
```
Доход = отработанные абонементы (списания за занятия) за месяц
Расход = расходы по периоду признания (см. ниже), не по дате платежа
ЗП = начисления преподавателям (по дате занятия) + окладники (Employee.monthly_salary × N месяцев) + bonus − penalty
Прибыль (убыток) = Доход − Расход − ЗП
```

**Период признания расхода (ExpenseRecognitionMode):**
| Режим | В ДДС | В ОПИУ |
|---|---|---|
| `by_payment_date` (default) | По `date` платежа | Месяц `date` |
| `single_period` | По `date` | Месяц `amortization_start_date` |
| `amortized` | По `date` (полная сумма) | 1/N в каждый из N месяцев начиная с `amortization_start_date` |

**Примеры:**
- Аренда июня 30 000 ₽, уплачена 25 мая → ДДС 25.05 −30 000; ОПИУ май: 0, ОПИУ июнь: 30 000.
- Принтер 30 000 ₽, `amortized` N=3 start=01.06 → ДДС 01.06 −30 000; ОПИУ июнь/июль/август по 10 000.
- Регрессия: расходы с `by_payment_date` (default для существующих записей) дают идентичный P&L.

**Реализация:** `app/src/lib/expense-amortization.ts` (`expandExpenseToMonths`, `expenseAmountInWindow`). Окно выборки расходов для P&L расширяется на `AMORTIZATION_LOOKBACK_MONTHS = 60` назад. Эта же логика применяется в `pnl-by-direction`, `pnl-group`, `financial-distribution`.

**Распределение расходов:**
- Расход на конкретное направление (через `ExpenseBranch.direction_id`, при `is_variable=true`) → 100% на это направление.
- Постоянный расход / переменный без direction → пропорционально выручке направлений (`distributeFixedExpenses`).

**ЗП в P&L:**
- Преподаватели → `Attendance.instructorPayAmount` за период, по `lesson.group.direction_id`.
- Окладники → `Employee.monthlySalary` × число месяцев окна, с `defaultDirectionId` или распределение пропорционально выручке если null.
- Корректировки → `SalaryAdjustment` (bonus +, penalty −).
- Источник истины для разнесения по направлениям — `SalaryPaymentItem.direction_id` (для документа выплаты).

**Прочие доходы (Payment без client_id, с income_category_id):** отображаются в P&L отдельным блоком после постоянных расходов, перед чистой прибылью, с **разбивкой по категориям** (источник — `IncomeCategory.name`). Входят в чистую прибыль и в рентабельность:
- `Маржа = Выручка − Переменные` (без прочих доходов)
- `Чистая прибыль = Выручка + Прочие доходы − Переменные − Постоянные − ЗП`
- `Рентабельность = Чистая прибыль / (Выручка + Прочие доходы)`

Учёт по дате платежа (`Payment.date`), `type IN ('incoming', 'transfer_in')`, refund исключаем.

**Важно:** Отчёт можно строить в любой момент, но финальные значения месяца — после закрытия периода (выплаты ЗП, отметки посещений, внесение расходов).

### 7.3. P&L на уровне группы (Формат C)

**Что показывает:** Прибыльность конкретной группы.

**Формула:**
```
Прибыль группы = Выручка группы − ЗП инструктора − Доля переменных расходов − Доля постоянных расходов
Доля переменных расходов = Переменные расходы филиала × (Кол-во занятий группы / Общее кол-во занятий филиала)
Доля постоянных расходов = Постоянные расходы филиала × (Выручка группы / Общая выручка филиала)
```

**Определения:**
- **Выручка группы** — отработанные (списанные) суммы по абонементам учеников группы
- **ЗП инструктора за группу** — начисления инструктору за занятия именно этой группы
- **Доля переменных расходов** — канцтовары/хозтовары со склада, пропорционально количеству занятий группы
- **Доля постоянных расходов** — пропорционально выручке группы к общей выручке филиала

**Производные метрики:**
- **Стоимость абонемента на ученика** = Выручка группы / Количество активных учеников
- **Рентабельность группы** = Прибыль группы / Выручка группы

**Пример:** Группа: 20 000₽ выручки из 30 000₽ общей. Аренда 9 000₽, канцтовары 3 000₽ (60 занятий в группе из 100 в филиале). Доля аренды = 9 000 × (20 000/30 000) = 6 000₽. Доля канцтоваров = 3 000 × (60/100) = 1 800₽. ЗП = 5 000₽. Прибыль = 20 000 − 5 000 − 1 800 − 6 000 = 7 200₽.

### 7.4. % распределения финансового результата

**Формула:**
```
% = Расходы (по статье) / Доходы
```

**Столбцы:** Статья расходов, Доходы (выручка), Чистая прибыль, Расходы (по каждой статье), %.

**Только после закрытия месяца.**

### 7.5. Сколько денег приносит педагог

**Формула:**
```
Прибыльность педагога = Выручка − Зарплата − Переменные расходы (канцтовары, пропорц. по занятиям) − Доля постоянных расходов
% от общего дохода = Прибыльность педагога / Общая чистая прибыль
```

**Столбцы:** Педагог, Выручка (отработанные), Зарплата (начисления), Переменные расходы (канцтовары), Постоянные расходы (пропорц.), Прибыльность, %.

**Баг в 1С:** Зарплата учитывает только начисления, премии и штрафы НЕ учитываются.

**Погрешность:** До 5% из-за усреднения расходов.

---

### 7.6. Сводный по абонементам в разрезе педагогов

**Столбцы:**
- **Активные абонементы** — хотя бы 1 списание
- **Новые** — дата продажи И первое занятие с явкой (не пробное) в выбранном месяце
- **Выбывшие** — не купили следующий абонемент (по дате последнего платного занятия)
- **Активные на конец месяца** — активные, оставшиеся на последнее число

**Баг в 1С:** Если продажа в одном месяце, а первое занятие в другом → не попадает в «Новые».

**Формула ROMI:**

- Наименование
- Количество
- Сумма (стоимость в наличии)

### 8.2. Канцтовары по дням

**Столбцы:**
- Филиал, место хранения, наименование
- Итого — приходы − расходы за месяц
- Конечный остаток — на последнее число
- Расход всего — за месяц

**Баг в 1С:** Конечный остаток не совпадает с отчётом «Остатки канцтоваров».

---

### 8.3. Сумма расходов по канцтоварам

**Столбцы:**
- Филиал, кабинет, наименование
- Количество израсходовано
- Стоимость за единицу
- Сумма расходов = стоимость за единицу × количество
- Количество активных абонементов за месяц (не учеников! ребёнок на 2 направлениях = 2 абонемента)
- **Средние расходы на абонемент** = Сумма расходов / Кол-во активных абонементов

**Альтернативная формула распределения (более точная):**
```
Расход на абонемент = (Сумма расходов филиала / Общее кол-во занятий в филиале) × Кол-во занятий абонемента
```
Эта формула учитывает, что ребёнок, отходивший 8 занятий, потребляет больше ресурсов, чем ребёнок с 3 занятиями.

## 10. Дополнительные отчёты и бизнес-правила (из базы знаний, стр. 3)

### 10.1. Доход от новых клиентов / упущенный доход по выбывшим (дополнение)

**Столбцы (уточнение):**
- **Количество новых** — клиенты с первым платным занятием в выбранном периоде. Клиент появляется при отметке хотя бы 1 занятия со списанием (первое платное или платное пробное) ИЛИ при внесении первой оплаты
- **Доход от новых** — сумма занятий со списаниями у новых клиентов в текущем периоде
- **Количество выбывших** — по дате последнего платного занятия в выбранном периоде (не по дате административного отчисления)
- **Факт упущенного дохода текущего месяца** — сумма занятий, которые клиент не доходил до конца месяца. Пример: 8 занятий × 100₽, отходил 6, выбыл → упущено 200₽
- **Упущенный доход следующего месяца (план)** — только при заполненном расписании. Берётся из «Средней стоимости абонемента» по филиалу и направлению за следующий месяц по активным абонементам

### 10.2. Индивидуальное расписание ученика

**Бизнес-правило:** Клиент может ходить в группу не на все дни (напр. группа Пн+Ср, клиент только по Пн). При уточнении расписания:
- Система удаляет ученика из неактуальных дней в текущем и будущих месяцах
- Стоимость абонемента пересчитывается автоматически
- Формирование занятий учитывает уточнённое расписание

### 10.3. Сегментация клиентов (расширенная логика)

**Критерии сегментации (три варианта):**
1. По количеству занятий
2. По сумме начислений
3. По количеству дней после первого платного занятия

**Правила:**
- У каждого сегмента есть **приоритет** (при попадании под несколько — выигрывает с высшим приоритетом)
- Сегментация применяется ко всей базе на **следующие сутки** после настройки
- Можно использовать готовые сегменты или создать новые

### 10.4. Цикличные задачи (расширенная логика)

**Поля:** описание, ответственный, приоритет (опц.), филиал и направление (опц.)

**Настройка расписания:**
- **Недельное:** выбор конкретных дней (напр. Пн-Ср-Пт)
- **Месячное:** выбор месяцев + день:
  - «Выполнять в N день с начала месяца» (1-е число, 5-е и т.д.)
  - «Выполнять в N день с конца месяца» (последний день)
  - «Выполнять в N-й день недели в месяце» (первый понедельник и т.д.)
- **Период:** дата начала + дата окончания (опционально)

### 10.5. Отчёт по посещениям

**Что показывает:** Выгрузка посещений за любой период по любому виду дня.

**Фильтры:** ФИО ученика, филиал, вид дня, период.

**Второй вид:** Количество каждого вида посещений по филиалам (свод).

### 10.6. Расчёты с педагогами

**Что показывает:** Расчёты по правилам бухучёта, актуален для бухгалтера/управляющего. Для руководителя актуальнее «Зарплата педагогов».

### 10.7. Потенциальные сотрудники

**Что показывает:** Список кандидатов на позиции (HR-модуль).

**Приоритет:** v2 (не MVP).

## 11. Общие принципы для реализации

1. **Каждый отчёт = кнопка «Справка»** — описание расчёта, определения столбцов, примеры
2. **Drill-down везде** — каждая цифра кликабельна → детализация до первичного документа
3. **Данные реального времени** — отчёты обновляются при каждом действии
4. **Фильтры** — филиал, направление, период, сотрудник (комбинируемые)
5. **Экспорт** — Excel/PDF с брендированием на каждом отчёте
6. **Исправить баги 1С** — все отмеченные баги должны быть исправлены в новой реализации
7. **Индивидуальное расписание** — клиент может ходить не на все дни группы, пересчёт абонемента автоматический

### Объединение отчётов
7 пар отчётов объединены в один с переключателем (решение 19.03.2026):
- Активные абонементы + Сводный по педагогам → группировка: филиал / педагог / общий
- Отток по педагогам + Отток по направлениям → группировка: педагог / направление
- Финрез + % распределения → один отчёт + переключатель "показать %"
- ЗП педагогов + Расчёты с педагогами → вид: руководитель / управляющий
- Остатки канцтоваров + Остатки → один отчёт
- Детализация оттока + Отток по направлениям → drill-down одного отчёта
- Поступления по дням + ДДС → drill-down из ДДС

### Обработка крайних случаев
- Деление на 0 (нет платежей, нет активных и т.д.) → показывать "—"
- Аномальные данные (клиент пришёл 30-го числа, 1 занятие) → не фильтруем, но не влияет на среднее при малой выборке
- Нулевая выручка всех направлений (начало сезона) → постоянные расходы распределяются равными долями
- Педагог с 0 активных учеников → не показывать в конверсии оттока

## 12. Учёт отработок (Ф7)

Cross-group отработка фиксируется через пару Attendance:

1. **На исходном (пропущенном) L1** — Attendance с `code=makeup_scheduled`, `scheduledMakeupLessonId=L2.id`. Не списывает, ЗП не начисляет.
2. **На целевом (отработанном) L2** — Attendance с `isMakeup=true`, `makeupOfLessonId=L1.id`, `subscriptionId=L1.sub.id`. Списание идёт с абонемента ИСХОДНОЙ группы, ЗП — педагогу L2.

**Влияние на отчёты:**
- Отчёт «Зарплата педагогов» (6.1) — отработка засчитывается педагогу L2.
- Отчёт «Активные абонементы» (1.1) — отработка считается списанием в месяце L2 (date), не L1.
- Отчёт «Отработанные абонементы» (5.10) — `chargedAmount` исходного абонемента растёт в момент отработки.
- Отчёт «Сверка актива» (1.7) — учитывается обе записи: maker_scheduled на L1 + makeup на L2.

**Если отработка не состоялась** (педагог поставил «Не был» или L2 отменён) — создаётся задача `missed_makeup` админу, `makeup_scheduled` на L1 остаётся. На отчёты влияет так же, как обычный `makeup_scheduled` — не списывает, не платит.

## 13bis. Особенности пакетного типа абонемента

`Organization.subscriptionType` определяет модель работы:
- `calendar` — абонемент привязан к календарному месяцу (`periodYear`, `periodMonth` обязательны).
- `package` — абонемент = N занятий со сроком годности (`startDate`, `expiresAt`).
- `fixed` — зарезервировано (в реализации не задействовано).

Тип выбирается в wizard онбординга и блокируется (`subscriptionTypeLockedAt`) после
создания первого абонемента. Разблокировка — только техподдержкой.

### 13bis.1. Поиск абонемента при отметке посещения

Для `subscriptionType = 'package'` поиск абонемента в attendance-route идёт по
**FIFO** среди пакетов, у которых:
- `startDate <= lesson.date`
- `expiresAt IS NULL OR expiresAt >= lesson.date`
- `balance > 0` (есть остаток денег = остаток занятий, т.к. списание идёт на `lessonPrice`)
- `status IN ('active', 'pending')`

`orderBy: startDate ASC` — самый старый пакет списывается первым.

### 13bis.2. Истечение пакета

Cron `close-expired-packages` (03:15 UTC ежедневно) переводит пакеты с
`expiresAt < today` в `status = 'closed'`, `endDate = today`. **Остаток balance
сгорает** (выручка остаётся у организации) — это решение бизнеса.

Lazy-защита: в attendance-route `expiresAt >= lessonDate` уже исключает
просроченные пакеты от списания, даже если cron не успел отработать.

Ручное продление (`PATCH /api/subscriptions/[id] { expiresAt }`): меняет дату,
а если пакет уже был закрыт по истечении — реактивирует (`status = 'active'`).

### 13bis.3. Уведомления о скором истечении

Cron `notify-expiring-packages` (06:00 UTC ежедневно) создаёт in-app уведомления
типа `package_expiring` для всех owner/manager/admin тенанта, у которого:
- `subscriptionType = 'package'`
- `packageExpiryNotifyDaysBefore > 0`

Условие выборки пакетов: `expiresAt IN (today, today + N]`, `balance > 0`.

Идемпотентность: уведомление по конкретному `Subscription.id` не создаётся
повторно в течение 24 часов.

### 13bis.4. Адаптация месячных отчётов

Все отчёты, фильтрующие по `(periodYear, periodMonth)`, для package-тенанта
используют **пересечение интервалов**:
```
startDate <= dateTo  AND  (expiresAt IS NULL OR expiresAt >= dateFrom)
```

- **Активные абонементы (1.1)** — package со `status IN (active, pending)` и пересечением периода.
- **Не продлённые** — пакеты, истёкшие в окне без нового пакета того же
  (clientId, directionId) со `startDate > expiresAt - 7 дней`.
- **Должники** — package с `balance > 0`, пересекающий выбранное окно.
- **Ожидаемые поступления** — `SUM(balance)` активных пакетов на `dateFrom`.

### 13bis.5. P&L и выручка

**Не требуют изменений.** Выручка считается через `Attendance.chargeAmount`
по дате занятия (см. 7.2), что корректно для обоих типов: для пакета
выручка естественным образом распределяется по месяцам отработки.

### 13bis.6. Шаблоны пакетов

`PackageTemplate` хранит пресеты (4/8/12 занятий и опциональный `validDays`).
При создании абонемента:
- Если `packageTemplateId` указан: `validDays = data.validDays || template.validDays || org.packageDefaultValidDays`.
- Если шаблон не выбран (ручной ввод): `validDays = data.validDays || org.packageDefaultValidDays`.
- `expiresAt = startDate + validDays`.

`groupId` остаётся обязательным — один пакет = одна группа.

## 13. Расчёт ЗП по новым схемам (Ф3)

Пять схем `SalaryScheme`. Применяется через `resolveRate` + `calcPay`:

1. **`per_student`** — `rate × count(students_present)`. Платится за каждого пришедшего.
2. **`per_lesson`** — `rate` (фикс за занятие), выплачивается **только на первой платной отметке** в lesson. Остальные отметки в том же lesson = 0 (чтобы не дублировать).
3. **`fixed_plus_per_student`** — исторически эквивалент `per_student` (поле `ratePerStudent` × число учеников). Сейчас в UI отображается «фикс за выход» как информация, но в расчёт идёт `ratePerStudent`.
4. **`percent_of_payments`** — процент от `chargeAmount` ТЕКУЩЕЙ отметки (списание этого ученика).
5. **`floating_by_students`** — плавающая матрица `SalaryBracket{minStudents, ratePerLesson}`. На основе текущего числа платных отметок в lesson выбирается bracket. Выплачивается **только на первой платной отметке** (как `per_lesson`).

**Приоритет выбора ставки:**
1. `GroupSalaryRate` для текущей группы — перекрывает ВСЕ личные ставки (включая замену).
2. Личное `SalaryRate` с `directionId = lesson.group.directionId` — исключение по направлению.
3. Личное `SalaryRate` с `directionId = null` — дефолт педагога.
4. Нет ставки → ЗП = 0, событие логируется в отчёт «педагоги без ставки».
