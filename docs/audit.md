## 1. Критические блокеры

Эти проблемы ЛОМАЮТ ключевую логику. Исправить до начала кодирования.

### 1.1. `Attendance.subscription_id` обязательное — ломает пробные занятия

> ✅ **Решено 19.03.2026:** subscription_id nullable, добавлен trial_lesson_id

**Где:** data-dictionary, таблица `Attendance`
**Проблема:** Поле `subscription_id: UUID, да (обязательное)`. Но на пробном занятии абонемента нет. `TrialLesson` — отдельная сущность, но посещение пробного тоже фиксируется в `Attendance`.
**Решение:** Сделать `subscription_id` nullable. Добавить `trial_lesson_id: UUID, nullable, FK → TrialLesson`. Constraint: одно из двух должно быть заполнено.

### 1.2. `Payment.client_id` обязательное — ломает выемки и инкассации

> ✅ **Решено 19.03.2026:** отдельная таблица AccountOperation для выемок/инкассаций

**Где:** data-dictionary, таблица `Payment`
**Проблема:** PRD (US-023, FIN-08): "Приход, расход, выемки собственника, инкассации". `PaymentType` включает `owner_withdrawal | encashment`, но это не платёж клиента, а операция владельца. `client_id` обязательное — нельзя создать выемку.
**Решение:** Либо `client_id` nullable, либо отдельная таблица `AccountOperation` для внутренних операций.

### 1.3. `SubscriptionStatus` нет статуса "ожидание оплаты"

> ✅ **Решено 19.03.2026:** добавлен статус pending в SubscriptionStatus

**Где:** data-dictionary, enum `SubscriptionStatus`
**Проблема:** PRD (US-004): "Абонемент создан, статус — Ожидание оплаты". Enum: `active | closed | withdrawn`. Нет `awaiting_payment` или `pending`.
**Решение:** Добавить `pending` в `SubscriptionStatus`.

### 1.4. Индивидуальное расписание (SCH-14) — нет модели данных

> ✅ **Решено 19.03.2026:** добавлено selected_days в GroupEnrollment

**Где:** PRD vs data-dictionary
**Проблема:** PRD (SCH-14): "Клиент ходит не на все дни группы (Пн+Ср — только Пн). Пересчёт абонемента автоматический". В DD нет ни одной таблицы/поля для хранения индивидуального расписания ученика в группе. `GroupEnrollment` не имеет выбранных дней.
**Решение:** Добавить таблицу `EnrollmentSchedule(enrollment_id, day_of_week)` или JSON-поле `selected_days` в `GroupEnrollment`.

### 1.5. `deleted_at` нигде не реализован

> ✅ **Решено 19.03.2026:** deleted_at добавлен во все ключевые сущности

**Где:** data-dictionary, соглашения vs таблицы
**Проблема:** В соглашениях: "Soft delete — поле `deleted_at`". Но ни одна из 42 таблиц не содержит `deleted_at`. Soft delete невозможен.
**Решение:** Добавить `deleted_at: DateTime, nullable` на все сущности, подлежащие мягкому удалению (Client, Employee, Group, Subscription, Direction, Room, и т.д.). Определить список.

## 2. Противоречия между документами

### 2.1. Пробные занятия — двойная модель

> ✅ **Решено 19.03.2026:** единая модель TrialLesson, убраны дублирующие флаги

**PRD (US-003):** Пробное может быть бесплатным или платным. При платном — стоимость засчитывается в абонемент.
**DD:** Есть `TrialLesson` (отдельная сущность) И `is_trial` в `Lesson` И `is_trial` в `Attendance` И `is_trial_credited` в `Subscription`. Но нет связи: как `TrialLesson.paid_amount` попадает в `Subscription.balance`? Нет `trial_lesson_id` в `Payment`.
**Рекомендация:** Убрать дублирование. Одна модель: `TrialLesson` хранит всё о пробном. `Attendance.trial_lesson_id` (nullable) для связи. `Payment.trial_lesson_id` (nullable) для оплаты пробного.

### 2.2. Копирование постоянных расходов (FIN-05)

> ✅ **Решено 19.03.2026:** флаг is_recurring + платёжный календарь

**PRD:** "Копирование постоянных расходов с прошлого месяца".
**DD:** В `Expense` нет `is_recurring` или `source_expense_id`. `PlannedExpense` покрывает плановые, но PRD описывает копирование фактических.
**Рекомендация:** Добавить `is_recurring: Boolean` + `recurring_group_id: UUID` в `Expense`.

### 2.3. Объединение дубликатов (CRM-12) — неполный каскад

> ✅ **Решено 19.03.2026:** каскадный перенос всех связей + мягкое удаление дубля

**PRD:** "Объединение дубликатов с превью".
**DD:** `ClientMergeLog` есть, но нет описания каскадного переноса: что с `Attendance`, `Payment`, `Subscription`, `CommunicationLog`? Все FK останутся на старом `client_id`.
**Рекомендация:** Описать миграцию FK при мерже. Или реализовать через обновление `client_id` во всех связанных таблицах + soft delete оригинала.

### 2.4. Баланс клиента — нет аудита

> ✅ **Решено 19.03.2026:** добавлена таблица ClientBalanceTransaction

**PRD (US-004):** "Остаток при отчислении переносится на баланс клиента".
**DD:** `Client.client_balance` — одно число. Нет истории операций.
**Рекомендация:** Добавить `ClientBalanceTransaction(client_id, amount, type, subscription_id, description, created_at)`.

### 2.5. Загруженность центра (CRM-34) — нет данных о часах кабинета

> ✅ **Решено 19.03.2026:** часы кабинета наследуются от филиала, можно переопределить

**PRD:** "Факт часов / макс часов по кабинетам".
**DD:** `Room` имеет `capacity` (чел.), но нет рабочих часов кабинета. `Branch.working_hours_start/end` — это филиал, не кабинет.
**Рекомендация:** Наследовать часы кабинета от филиала (по умолчанию) или добавить `working_hours` в `Room`.

## 7. Мультитенант: риски утечки данных

### 7.1. Справочники с `tenant_id = null` (системные)

> ✅ **Решено 20.03.2026:** стратегия RLS описана — системные справочники через WHERE tenant_id = current OR tenant_id IS NULL. Тесты изоляции обязательны при разработке каждой таблицы

`AttendanceType`, `ExpenseCategory`, `WithdrawalReason`, `AbsenceReason`, `MarketingChannel` — все имеют `tenant_id = null` для системных записей. RLS-политика должна возвращать `WHERE tenant_id = current ИЛИ tenant_id IS NULL`.

**Риск:** Одна ошибка — и пользовательские справочники одного тенанта видны другому.
**Рекомендация:** Тесты на изоляцию тенантов для каждого справочника. Или отдельная таблица `SystemDefault*` без tenant_id.

> ✅ **Решено 19.03.2026:** экран выбора организации

**Рекомендация:** Экран выбора организации после логина (как в Slack).

### 7.3. `Invoice` и `SupportTicket` — без tenant_id

> ✅ **Решено 20.03.2026:** Invoice и SupportTicket используют organization_id, RLS-политики пишутся отдельно от tenant-уровневых

Используют `organization_id` вместо `tenant_id`. RLS-политики нужно писать отдельно.

## 9. Инсайты из custdev

### 9.1. Собственники НЕ пользуются финансовой аналитикой

Семенюк: "финансовые штуки понравились, но я до сих пор ничем не пользуюсь". Лера: финрез не закрывает.
**Вывод:** Финансовый модуль ценен, но ТОЛЬКО если данные корректно вносятся. Нужен максимально простой UX для ввода расходов.

### 9.2. Решения принимаются по воронке, не по финансам

Лера: рост = "лиды → пробники → покупки + иногда отток по педагогам".
**Вывод:** Воронка и конверсии — приоритет №1 для UX. Финрез — "nice to have" для продвинутых.

### 9.3. AlphaCRM проигрывает в поддержке

Семенюк: "AlphaCRM такие большие, им вообще пофигу".
**Вывод:** Человеческая поддержка — конкурентное преимущество. Сохранять.

### 9.4. Педагоги вообще не работают в CRM

Ксения/Палевич и Лера подтвердили: педагоги не заходят, посещаемость отмечают админы.
**Вывод:** Роль инструктора в MVP может быть минимальной (только расчётный листок).

### 9.5. Lock-in через CRM слабый

Семенюк: если CRM пропадёт, "болезненно, но можно перенести".
**Вывод:** Создавать ценность, которую сложно перенести: аналитика трендов, автоматизация.

### 9.6. Справки задним числом ломают зарплату

PRD (SAL-11b) описывает корректировки — ОК, но нужно сделать максимально наглядно.

## 11. Архитектурные риски

### 11.1. Каскад при отметке посещения

> ✅ **Решено 19.03.2026:** транзакции

Одна отметка: `Attendance` → `Subscription.balance` → `SalaryAccrual` → `Client.client_balance`. Массовая отметка (15 учеников) = 60 операций.
**Рекомендация:** Транзакционная стратегия + очередь для начислений ЗП (async).

### 11.2. RLS + сложные JOIN = медленные отчёты

> ✅ **Решено 20.03.2026:** materialized views будут созданы для тяжёлых отчётов (прибыльность педагога, финрез по группам). Стратегия: обновление по расписанию (ночной job) + принудительное при закрытии периода

"Прибыльность педагога": 5 таблиц с RLS на каждой.
**Рекомендация:** Materialized views или денормализованные таблицы для отчётов.

### 11.3. Денормализованные балансы без reconciliation

> ✅ **Решено 19.03.2026:** фоновый пересчёт + отчёт "Сверка"

`Account.balance`, `Client.client_balance`, `Subscription.balance` — любой баг ломает баланс.
**Рекомендация:** Фоновый job пересчёта. Отчёт "Сверка" для админа.

### 11.4. NextAuth.js + кастомная Employee — непонятная связь

> ✅ **Решено 19.03.2026:** email для владельца, логин/пароль для сотрудников, без NextAuth adapter

NextAuth создаёт свои таблицы. DD описывает `Employee.password_hash`. Как связаны?
**Рекомендация:** Определить: NextAuth с кастомным adapter → Employee, или свой auth.

### 11.5. Два статуса Client — риск рассинхронизации

> ✅ **Решено 19.03.2026:** триггер без возврата, из лида в актив при первой оплате, обратно никогда

`funnel_status` и `client_status` — параллельные. Нет описания триггера перехода.
**Рекомендация:** Описать state machine с триггерами.

### 11.6. Production Calendar per tenant — overhead

> ✅ **Решено 19.03.2026:** системный календарь + исключения партнёра

Каждый из 500 партнёров будет иметь свою копию календаря. 95% — один и тот же.
**Рекомендация:** Системный календарь + пользовательские исключения.
