## 1. Обзор продукта

### 1.1. Что это

Веб-система управления бизнесом в сфере услуг: детские центры, частные школы, студии (танцы, фитнес, растяжка), салоны красоты, массажные кабинеты и т.д.

Четыре слоя функциональности:
- **CRM-слой** — воронка продаж (лиды), работа с клиентской базой (допродажи, возвраты), конверсии, каналы привлечения, обзвон
- **Операционный слой** — расписание, группы, посещения, абонементы, склад
- **Финансовый слой** — оплаты, расходы, зарплаты, P&L, ДДС, кассы, онлайн-эквайринг
- **SaaS-слой** — бэк-офис, биллинг, личные кабинеты (партнёра и клиента)

### 1.2. Предыстория

Система существует с 2023 года на платформе 1С (Smart CRM). 20 клиентов (организаций), ~60-100 конечных пользователей. Модель SaaS, подписка 5-7 тыс. руб./мес.

Причины перезапуска на вебе:
- Сложное внедрение (высокий порог входа для администраторов)
- Дорогая разработка и поддержка на 1С
- Кривая воронка продаж (дубли, некорректные отчёты)
- Невозможность адаптировать под пакетные и разовые услуги

## 2. Технологический стек

| Компонент | Технология | Обоснование |
|---|---|---|
| Язык | TypeScript | Один язык на фронт и бэк, типобезопасность |
| Фреймворк | Next.js (App Router) | Фулстек в одном проекте, SSR, API routes |
| БД | PostgreSQL | Надёжна для финансовых данных, Row Level Security для мультитенанта |
| ORM | Prisma | Типобезопасная работа с БД, миграции из коробки |
| UI | shadcn/ui + Tailwind CSS | Готовые компоненты, быстрая сборка интерфейса |
| Мобилка | PWA (next-pwa) | Без отдельного мобильного проекта |
| Авторизация | NextAuth.js | Роли, сессии, мультитенант |
| Деплой | Docker → VPS | Контроль над данными, предсказуемая стоимость |
| Онлайн-оплата | ЮKassa + Робокасса | Партнёр подключает свой аккаунт через API-ключи |

### 2.1. Мультитенантность

Схема **A**: одна БД, одна схема, `tenant_id` в каждой таблице + Row Level Security (RLS) на уровне PostgreSQL. Масштабируется на 500+ партнёров.

### 2.2. Модель авторизации

- **Основной аккаунт (владелец)** регистрируется по email — это единственный аккаунт с email
- **Сотрудники** (администраторы, инструкторы и т.д.) — владелец создаёт им логины (латиница) + пароли (как в 1С). Email сотрудникам **не нужен**
- **Вход:** логин + пароль (простая форма, без OAuth/соцсетей)
- **Восстановление пароля** — только для аккаунта с email (владелец). Сотрудники обращаются к владельцу

## 3. Глоссарий

> Единые термины для всей команды. Согласовано устно 10.02.2026, подтверждено 15.03.2026.

| Термин | Определение | Старый термин в 1С |
|---|---|---|
| **Лид** | Контакт на этапе воронки продаж (до первой оплаты абонемента). НЕ является клиентом. | Лид, Контрагент |
| **Клиент** | Физическое лицо, которое платит за услуги. Становится клиентом после первой оплаты. В детском центре — родитель. В фитнес-студии — сам посетитель. | Контрагент, Плательщик |
| **Подопечный** | Тот, кто получает услугу (если отличается от клиента). В детском центре — ребёнок. Информационная сущность — не участвует в финансовой логике. | Ученик, Ребёнок |
| **Абонемент** | Предоплата за определённое количество определённых услуг. Привязан к клиенту. Каждый месяц — новый абонемент (важно для LTV). | Договор, Направление |
| **Направление** | Тип услуги: "Английский", "Скорочтение", "Стрейчинг". Определяет базовую стоимость и длительность. | Направление деятельности |
| **Группа** | Конкретный слот в расписании: направление + кабинет + время + инструктор + состав учеников. | Группа обучения |
| **Занятие** | Один конкретный урок/сеанс в расписании. Имеет дату, время, группу, инструктора. | Занятие |
| **Посещение** | Факт присутствия или отсутствия клиента/подопечного на занятии. Имеет тип (вид дня). | Посещение |
| **Списание** | Снятие суммы с баланса абонемента клиента за одно занятие (при явке или прогуле). | Списание |
| **Перерасчёт** | Занятие без списания (уважительная причина, справка). Потеря выручки для центра. | Неявка уважительная |
| **Отработка** | Занятие без списания, но с пометкой, что ребёнку нужно допзанятие. | — (новое) |
| **Филиал** | Физический адрес/локация бизнеса. | Подразделение |
| **Кабинет** | Помещение внутри филиала, где проводятся занятия. | Кабинет |
| **Инструктор** | Сотрудник, проводящий занятия. Техническая роль — «инструктор». **Отображаемое название роли настраиваемо** организацией: «тренер», «педагог», «преподаватель» и т.д. | Преподаватель, Педагог |
| **Оборот** | Сумма всех поступлений денежных средств за период (ДДС). Когда деньги пришли. | Приход |
| **Выручка** | Сумма всех отработанных (списанных) занятий за период. Когда услуга оказана. | Доход |
| **Маржа** | Выручка минус переменные расходы (ЗП инструкторов + канцтовары/хозтовары со склада + прочие материалы). | — |
| **Рентабельность** | Чистая прибыль / выручка, в процентах. | — |
| **Финрез** | Финансовый результат: Выручка − Расходы = Прибыль/Убыток. | Финансовый результат |
| **Партнёр** | Организация (юрлицо/ИП), которая покупает подписку на нашу CRM. НЕ путать с «клиентом» — клиент существует внутри системы партнёра. | — (уровень SaaS) |
| **Средний чек** | Сумма всех оплат / количество платежей. Уровень ДДС, не связан с абонементами. Один платёж = один чек (даже если покрывает несколько абонементов) | — |
| **Средняя стоимость абонемента** | Сумма отработанных сумм / количество активных абонементов за месяц. Активный абонемент = хотя бы одно списание в течение месяца. Отработанная сумма = фактически списано (при уходе в середине месяца — только отработанная часть) | — |
| **Отток (дата)** | Считается по дате **последнего платного занятия**, а не по дате заявления клиента или административного отчисления. Если последнее занятие 25 мая, а клиент сообщил 5 июня — это отток мая. **Выбывший** — клиент, не купивший следующий абонемент (по факту прекращения оплат). «Отчисленный» администратором — отдельное действие, в отчётах оттока не используется | — |
| **Дата продажи** | Дата, когда клиент стал активным: **дата внесения денег** ИЛИ **дата первого платного занятия** (если пришёл в долг, без оплаты). Что наступило раньше. Статус «Ожидание оплаты» — внутренний для админа, **не попадает ни в какой отчёт** | — |
| **Счёт (кошелёк)** | Место хранения денег: касса наличных, расчётный счёт, эквайринг-терминал, онлайн-эквайринг. Произвольное количество, привязка к филиалу/кабинету/организации. | Касса |
| **Заявка** | Запрос лида на конкретное направление/абонемент. У одного лида может быть несколько заявок на разные направления. Понятие «интерес лида» **не используется** | — |
| **Кандидат (потенциальный сотрудник)** | Кандидат на должность, находится в модуле сотрудников. ФИО, контакты, резюме (PDF), комментарий, этап, результат | — |

## 4. Карта сущностей

### 4.1. Диаграмма связей

```
Организация (tenant)
├── Филиал (1..N)
│   ├── Кабинет (1..N)
│   ├── Часы работы, рабочие дни недели (для отчёта загруженности)
│   └── Склад (товары на складе филиала)
│
├── Счёт / Кошелёк (1..N) — произвольное количество любого типа
│   ├── Тип: касса наличных / расчётный счёт / эквайринг / онлайн-эквайринг
│   └── Привязка: к филиалу, кабинету или организации целиком
│
├── Направление (1..N)
│   └── Базовая стоимость занятия, стоимость пробного, длительность (мин)
│
├── Сотрудник (1..N)
│   ├── Роль (владелец, управляющий, администратор, инструктор, только чтение)
│   ├── Тип: действующий / кандидат
│   ├── Личное дело: ФИО, контакты, ДР, дата начала работы
│   ├── Документы (загрузка PDF: паспорт, трудовой договор, СНИЛС, резюме и т.д.)
│   ├── Кандидат: статус (новый/собеседование/тестовый/принят/отказ), история встреч
│   ├── История выплат ЗП
│   ├── Список групп (для инструкторов)
│   └── Ставка ЗП (0..N) — привязана к сотруднику + филиалу + направлению + группа (опц.)
│       Окладная ЗП может быть без привязки к филиалу/направлению
│
├── Группа (1..N)
│   ├── → Филиал, Кабинет, Направление, Инструктор
│   ├── Шаблон расписания (дни + время для каждого дня + длительность)
│   ├── Лимит учеников
│   └── Занятие (генерируется из шаблона)
│       ├── Тема занятия (заполняет инструктор, опционально)
│       ├── Домашнее задание (заполняет инструктор, опционально)
│       └── Посещение (по каждому ученику)
│           ├── Тип (вид дня)
│           ├── Сумма списания
│           ├── Начисление ЗП инструктору
│           └── Переключатель: оплата инструктору за это занятие (вкл/выкл)
│
├── Лид / Клиент (1..N)
│   ├── ФИО (опц.), телефон (опц.), соцсети (опц.)
│   │   └── Обязательно хотя бы одно из: телефон ИЛИ соцсети
│   ├── Доп. телефон, email, канал привлечения (заполняется позже)
│   ├── Сегмент (авто: Новый / Стандарт / Постоянный / VIP)
│   ├── История коммуникации (лента: кто, когда, комментарий, дата след. контакта)
│   ├── Подопечный (0..N) — информационные поля (имя, возраст, ДР)
│   ├── Абонемент (0..N)
│   │   ├── → Направление, Группа, Подопечный (опц.)
│   │   ├── Тип (календарный / фикс / пакетный)
│   │   ├── Баланс, стоимость занятия, скидки (0..N)
│   │   └── Статус (активный / закрыт)
│   ├── Баланс клиента (общий, остатки с закрытых абонементов)
│   │   └── Журнал операций по балансу (дата, сумма, тип, комментарий) — для аудита
│   ├── Воронка лидов (для новых контактов)
│   │   └── Новый → Назначено пробное → Посетил пробное → Ожидание оплаты → Активный клиент
│   │       Плюс: Потенциал, Нецелевой, Чёрный список, Архив
│   ├── Работа с клиентской базой (для действующих/бывших)
│   │   └── Активный, Допродажа, Выбывший → Возврат, Архив
│   └── LTV (по деньгам + по количеству абонементов = месяцев)
│
├── Расход (1..N)
│   ├── Сумма, дата, статья расхода
│   ├── → Филиал (один/несколько/все), Кабинет, Направление
│   ├── Период амортизации (опц.): ДДС = полная сумма, финрез = 1/N помесячно
│   ├── Повторяющийся (опц.): автодублирование ежемесячно, отображение в календаре платежей
│   └── → Счёт
│
├── Склад
│   ├── Товар на складе: наименование (канцтовары, хозтовары и т.д.), количество, стоимость за единицу, сумма
│   ├── Перемещение: склад → кабинет (переменный расход в финрез в момент перемещения, по стоимости за единицу)
│   ├── Баланс кабинета: что лежит в кабинете, кнопка «Списать»
│   └── Амортизация при закупке: размазать расход на N месяцев
│
├── Модуль обзвона
│   ├── Фильтр базы (статус, возраст, филиал) → список обзвона
│   ├── В списке: номер, база-источник, дата связи, комментарий, ДР, интерес
│   ├── Результаты → история коммуникации клиента
│   └── Закрытый обзвон → архив
│
├── Модуль задач
│   ├── Ручные задачи (руководитель → админу)
│   ├── Автозадачи (из дат: оплата, связь, пробник, ДР, пропуски)
│   └── Приоритет по близости даты
│
├── Справочники (настраиваемые, предзаполненные)
│   ├── Виды дней: Явка, Прогул, Перерасчёт, Отработка + пользовательские
│   ├── Статьи расходов (14 предустановленных)
│   ├── Причины отчисления (7 предустановленных)
│   ├── Причины пропусков (7 предустановленных)
│   ├── Каналы маркетинга (13 предустановленных)
│   └── Скидки: постоянные / разовые / связанные
│
└── Настройки организации
    ├── Оплата инструктору за прогул ученика (да/нет)
    ├── Срок и лимит отработок
    ├── Дедлайн отметки посещений (дефолт: 14 дней)
    ├── Лимит отрицательного баланса клиента (без лимита / с лимитом + уведомление админу при долге)
    └── Настройка прав ролей (владелец конфигурирует)
```

### 4.2. Ключевые правила связей

1. **Лид → Клиент**: автоматически при первой оплате абонемента (ручного перевода не требуется). **Обратный переход в лида НЕВОЗМОЖЕН** — триггер без возврата
2. **Клиент → Абонементы**: один клиент — N абонементов. Каждый месяц — новый абонемент
3. **Абонемент → Группа**: один абонемент привязан к одной группе
4. **Группа → Инструктор**: в каждый момент один инструктор, замена возможна (ЗП по ставке заменяющего)
5. **Занятие → Посещения**: одно занятие = N посещений (по числу учеников)
6. **Посещение → Списание + ЗП**: зависит от вида дня и настроек организации
7. **Расход → Распределение**: постоянные расходы автораспределяются пропорционально выручке
8. **Воронка + клиентская база**: раздельные потоки, без смешения маркетинговой атрибуции
9. **Остаток при отчислении**: переносится на баланс клиента (не абонемента). При новом абонементе — система предлагает использовать остаток

---

## 5. Роли пользователей

> **Пять ролей.** Права ролей настраиваемые владельцем. **Отображаемые названия ролей** настраиваемы (например, «инструктор» → «тренер», «педагог»). Техническая роль сохраняется.

### 5.1. Владелец (= Руководитель / Собственник)

- Полный доступ ко всему
- Настройка системы (филиалы, кабинеты, направления, цены, ставки)
- Финансовые отчёты (P&L, маржа, ДДС)
- Управление сотрудниками и настройка прав ролей
- Чёрный список (с обязательной причиной) — всегда может
- **Закрытие и переоткрытие периодов**
- **Корректировки после закрытия периода**

### 5.2. Управляющий

- Права **настраиваются владельцем** индивидуально
- Сюда же входит роль «бухгалтера» — финансовый управляющий
- Обычно: доступ к финансам, отчётам, управлению сотрудниками
- Чёрный список — если разрешено владельцем
- **Корректировки после закрытия периода** — если разрешено владельцем

### 5.3. Администратор (= Менеджер)

- Работа с воронкой продаж, приём оплат, внесение расходов
- Доступ к клиентской информации, допродажи
- Отчёт пропусков → звонки клиентам
- Доступ к филиалам: настраиваемый набор
- **НЕ может** корректировать данные после закрытия периода
- **НЕ видит** (по умолчанию): зарплаты, финрез. Владелец может расширить доступ

### 5.4. Инструктор

- Своё расписание и состав групп
- Отметка посещений в своих группах
- Просмотр своей ЗП (настраиваемо)
- Запись пройденной темы и домашнего задания в занятии
- **Видит**: ФИ детей, возраст, ДР
- **НЕ видит**: контакты клиентов, балансы, зарплаты других, финансы
- **НЕ может** корректировать данные после закрытия периода

### 5.5. Только чтение

- Просмотр отчётов и дашбордов (набор настраивается владельцем)
- Для инвесторов, партнёров, внешних консультантов
- **НЕ может** редактировать данные

## 6. User Stories

### 6.1. Воронка продаж (CRM-слой)

#### US-001: Создание лида при входящем звонке
**Как** администратор,
**я хочу** быстро внести данные нового контакта,
**чтобы** не потерять контакт и начать продажу.

**Поля формы:**
- ФИО (родителя) — опционально
- Телефон — опционально (проверка на дубли и чёрный список)
- Соцсети (ссылка VK / Telegram / другое) — опционально
- **Обязательно хотя бы одно из**: телефон ИЛИ соцсети
- Канал привлечения — опционально
- Желаемое направление, филиал — опционально
- Имя и возраст подопечного — опционально
- Комментарий — опционально

**Быстрое создание:** кнопка «+» на главном экране.

**Защита:** При совпадении телефона — предупреждение. Если номер в чёрном списке — блокировка с причиной.

---

#### US-002: Назначение пробного занятия
**Как** администратор,
**я хочу** записать лида на пробное в конкретную группу,
**чтобы** он попробовал и купил абонемент.

**Действия:**
1. Открыть карточку лида
2. Выбрать группу (система показывает свободные места)
3. Выбрать дату пробного
4. Подтвердить

**Результат:** Статус → "Назначено пробное". Лид может иметь несколько пробных на разных направлениях.

**Автозадача:** За 24 часа → "Отправить напоминание".

---

#### US-003: Фиксация результата пробного
**Как** администратор,
**я хочу** отметить результат пробного,
**чтобы** двигать лида по воронке.

**Варианты:**
- **Пришёл** → статус "Посетил пробное"
- **Не пришёл** → задача "Перезвонить — не пришёл"
- **Отменил** → "Потенциал" с датой следующего контакта

**Оплата пробного:**
- **Бесплатное** (настройка по направлению) — ничего не списывается, инструктору не начисляется (по умолчанию). На конкретном занятии можно вручную включить оплату инструктору
- **Платное** — стоимость из настроек направления, ЗП инструктору начисляется
- Если после платного пробного клиент купил абонемент — стоимость пробного засчитывается

---

#### US-004: Продажа абонемента
**Как** администратор,
**я хочу** оформить покупку абонемента,
**чтобы** клиент стал активным и попал в расписание.

**Действия:**
1. Открыть карточку лида
2. «Создать абонемент» → направление, группа, подопечный
3. Система показывает: стоимость, кол-во занятий, сумму
4. Применить скидку (опционально)
5. Подтвердить

**Результат:** Абонемент создан, статус → "Ожидание оплаты".

---

#### US-005: Приём оплаты
**Как** администратор,
**я хочу** внести оплату от клиента,
**чтобы** баланс обновился и клиент стал активным.

**Действия:**
1. Открыть карточку → "Принять оплату"
2. Сумма, способ (наличные / безнал / эквайринг / онлайн / СБП), счёт
3. Привязать к абонементу

**Результат:** Статус → "Активный клиент" **автоматически** при одном из двух событий: 1) первая оплата, или 2) посещение первого платного занятия (без оплаты, «в долг»). Лид покидает воронку. Ручного перевода не требуется. **Статус «Ожидание оплаты»** — внутренний для админа (напоминание), не попадает ни в какой отчёт как «продажа».

**Способы оплаты:** наличные, безнал (перевод), эквайринг (терминал), онлайн (ЮKassa / Робокасса), СБП (QR-код).

---

#### US-006: Работа с воронкой (ежедневный ритуал)
**Как** администратор,
**я хочу** утром видеть список задач,
**чтобы** никого не потерять.

**Дашборд "Задачи на сегодня":**
- Лиды с датой связи = сегодня → "Позвонить"
- Пробные на завтра → "Отправить напоминание"
- Пробные сегодня → "Подтвердить"
- Пробные вчера без результата → "Узнать результат"
- "Ожидание оплаты" > 3 дней → "Напомнить"
- ДР подопечных сегодня → "Поздравить"
- Пропуски за вчера → "Уточнить причину"

**Сортировка лидов:** по дате следующего контакта (сегодняшние сверху). Заменяет предзапись.

---

#### US-007: Статусы воронки лидов

| Статус | Описание | Кто может |
|---|---|---|
| Новый лид | Первичный контакт | Админ |
| Назначено пробное | Записан на пробное | Админ |
| Посетил пробное | Был на пробном | Админ |
| Ожидание оплаты | Согласился купить | Админ |
| **→ Активный клиент** | Оплатил | Автоматически |
| Потенциал | "Позвоните позже" + дата связи | Админ |
| Нецелевой | Мусорный контакт | Админ |
| Чёрный список | Запрет + причина | Настраиваемое право (владелец всегда) |
| Архив | Потеряли контакт | Админ |

---

#### US-008: Допродажа и возврат клиента
**Как** администратор,
**я хочу** оформить допродажу или возврат клиента,
**чтобы** наращивать LTV без дублирования в воронке лидов.

- **Допродажа**: второе направление → не попадает в воронку лидов, не в маркетинговой аналитике, **но учитывается в бонусе админа**
- **Возврат**: выбывший → новый абонемент (не восстановление старого)

---

### 6.2. Операционный слой

#### US-010: Отметка посещений
**Как** инструктор или администратор,
**я хочу** отметить посещения,
**чтобы** рассчитались списания и ЗП.

**Действия:**
1. Открыть занятие → список учеников
2. Кнопка **"Отметить всех"** (явка всем) → убрать отсутствующих
3. Или вручную: Явка / Прогул / Перерасчёт / Отработка / другой вид

**Автоматически:**
- Списание с абонемента (по настройкам вида дня)
- Начисление ЗП инструктору (по настройкам вида дня + настройка «оплата за прогул» на уровне организации)

**Отметка задним числом:** в пределах настроенного дедлайна (по умолчанию 14 дней, настройка организации).

---

#### US-011: Просмотр расписания

**Основной вид — неделя** (7 дней).

**Виды:** по кабинетам, по инструкторам, по группам, по направлениям.

**Фильтры:** филиал, направление, инструктор, кабинет — комбинируемые.

**Цветовая индикация заполняемости:** зелёный (100%), жёлтый (51-99%), голубой (<50%), серый (нет учеников).

**Печать расписания:** Кнопка «Печать» на экране расписания → версия для принтера (без навигации, крупный шрифт, только расписание педагога на неделю). Для педагогов, которые не работают в CRM.

---

#### US-012: Создание группы и генерация расписания

**Поля группы:** название, направление, филиал, кабинет, инструктор, лимит учеников, шаблон (дни + время + длительность).

**Генерация расписания:**
- Помесячная из шаблона (можно запустить на несколько месяцев сразу)
- **Сентябрь** — всегда с нуля: новый шаблон на учебный год → генерация
- После генерации → создание абонементов
- Корректировка: удалить/закрыть группу, удалить отдельные занятия (праздники)

**Закрытие группы:** архивирование, не участвует в генерации.

---

#### US-013: Замена инструктора

- **Разовая**: на конкретное занятие, ЗП → заменяющему по его ставке
- **Постоянная**: с определённой даты, все будущие занятия → новому

---

#### US-014: Календарный абонемент

1. Стоимость занятия по направлению × кол-во занятий до конца месяца
2. Применение скидок
3. Каждое посещение → списание
4. Каждый месяц — **новый абонемент**
5. Клиент может начать в середине месяца

---

#### US-015: Закрытие периода

**Мягкое закрытие:** руководитель (владелец) закрывает месяц → фиксируется «снимок» финреза.

**После закрытия:**
- Администратор и инструктор — **не могут** вносить изменения
- Владелец и управляющий — **могут** корректировать (с аудитом, пометка «после закрытия»)
- Все корректировки логируются

**Переоткрытие:** владелец может переоткрыть период (справка задним числом).

---

#### US-016: Отработки

- Пропущенное занятие → разовое занятие в другой группе
- Списание — с действующего абонемента основной группы
- **Срок и лимит** отработок — настройка организации
- **Коэффициент отработок (1:2) НЕ делаем.** Отработка бьётся по деньгам, а не по часам. Решение 19.03.2026

---

### 6.3. Финансовый слой

#### US-020: Внесение расхода

**Простая форма:**
1. Сумма
2. Статья расхода (из справочника)
3. Филиал (один/несколько/все)
4. Оплачено из (счёт)
5. Распределить на N месяцев? (опционально)
6. Комментарий (опционально)

При распределении: в ДДС — полная сумма сразу, в финрезе — равными долями.

---

#### US-021: Должники

**Таблица:** филиал, направление, клиент, подопечный, плановый долг, фактический долг, дата последней оплаты, **обещанная дата оплаты**, комментарий.

**Подсветка:** просроченная обещанная дата — красным.

**Автозадача:** обещанная дата = сегодня → "Позвонить".

---

#### US-022: Финансовый результат

> **Финрез = Выручка − Расходы текущего периода** (с учётом амортизации).

**Формат A — общий P&L:**
```
Выручка (отработанные абонементы):     500 000
─────────────────────────────────────────────
Переменные расходы:
  Зарплата инструкторов:              -150 000
  Канцтовары и хозтовары (со склада): -15 000
  Прочие материалы:                    -5 000
─────────────────────────────────────────────
Маржа:                                 330 000 (66%)
─────────────────────────────────────────────
Постоянные расходы:
  Аренда:                             -80 000
  Маркетинг:                          -30 000
  ...
─────────────────────────────────────────────
Чистая прибыль:                       190 000
Рентабельность:                           38%
```

**Формат B — по направлениям/филиалам** (постоянные расходы пропорционально выручке).

**Drill-down:** каждая сумма кликабельна → расшифровка до первичных документов.

---

#### US-023: ДДС (полноценный)

- Приход, расход, выемки собственника, инкассации
- Сверка остатков по каждому счёту/кассе
- По каждому филиалу отдельно
- Перемещение между счетами

---

#### US-024: Зарплата

**Три схемы ЗП инструкторов:**
1. **За ученика**: фикс × кол-во учеников (настройка: платить за прогул — на уровне организации)
2. **За занятие (фикс)**: фиксированная сумма, не зависит от учеников
3. **Фикс за выход + за ученика**: базовая + доплата за каждого

**Привязка ставки:** сотрудник + филиал + направление + группа (опц.) + дата начала. Возможность вручную изменить ставку на конкретное занятие.

**Мотивация админа:** оклад/ставка + настраиваемый бонус за продажи (каналы, условия, сумма за продажу).

**Сводка:** начислено / премии / штрафы / выплачено / осталось. Детализация до занятия.

**Корректировки прошлых периодов:** отображаются в текущем месяце с провалом в источник.

---

#### US-025: Склад и канцтовары

1. **Закупка** → товар на склад (наименование, количество, стоимость за единицу, сумма). В ДДС — полная сумма в момент покупки
2. **Перемещение** → склад → кабинет, **переменный расход в P&L** в момент перемещения (по стоимости за единицу × количество). Списание поштучно (не всей партии сразу)
3. **Баланс кабинета** → что лежит, кнопка «Списать»
4. **Амортизация** → размазать на N месяцев

---

## 13. План миграции

### 13.1. Что переносим

| Данные | Источник | Обработка |
|---|---|---|
| Клиенты (родители) | Выгрузка из 1С | Дедупликация, нормализация ФИО, структурирование (клиент → подопечные) |
| Остатки балансов | Выгрузка из 1С | Долг/переплата по каждому клиенту-направлению |
| Остатки денег | Ручной ввод | Начальные остатки касс и счетов |

### 13.2. Что НЕ переносим

- Инструкторы и сотрудники (< 20, вводятся руками)
- Расписание (создаётся заново)
- История оплат и посещений
- Воронка продаж (начинаем с нуля)

### 13.3. Способ миграции

**Скрипт миграции напрямую из 1С.** Клиент ничего не делает — мы выгружаем и заливаем. Единая версия 1С у всех 20 партнёров.

### 13.4. Процесс

Сроки — по ситуации (пилот в апреле, массовый переезд по готовности). Этапы:
1. Предупреждение партнёров
2. Настройка системы (справочники через wizard)
3. Импорт данных (скрипт)
4. Параллельная работа (2 недели адаптация)
5. Отключение 1С
6. Сбор обратной связи

### 14.1. MVP

**Цель:** полноценная CRM для детских центров, переезд 20 партнёров.

**Модули:**
- CRM: воронка, клиентская база, обзвон, сегментация, история коммуникации, детализация оттока
- Расписание: группы, занятия, посещения, закрытие периода, отработки
- Абонементы: календарный тип, скидки (3 типа), баланс, возврат

- Зарплата: 3 схемы инструкторов + мотивация админа, карточка сотрудника, кандидаты
- Склад: закупка → перемещение → баланс кабинета → списание
- Дашборд: настраиваемый по ролям + задачи (ручные + авто)
- Обзвон: отдельный модуль
- Личные кабинеты: партнёра + клиента
- Бэк-офис: биллинг, управление партнёрами
- Администрирование: 5 ролей, мультитенант (RLS), wizard, аудит
