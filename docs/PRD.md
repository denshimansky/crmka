## 1. Обзор продукта

### 1.1. Что это

Веб-система управления бизнесом в сфере услуг: детские центры, частные школы, студии (танцы, фитнес, растяжка), салоны красоты, массажные кабинеты и т.д.

Три слоя функциональности:
- **CRM-слой** — воронка продаж (лиды), воронка клиентов (допродажи, возвраты), конверсии, каналы привлечения
- **Операционный слой** — расписание, группы, посещения, абонементы
- **Финансовый слой** — оплаты, расходы, зарплаты, P&L, точка безубыточности

### 1.2. Предыстория

Система существует с 2023 года на платформе 1С (Smart CRM). 20 клиентов (организаций), ~60-100 конечных пользователей. Модель SaaS, подписка 5-7 тыс. руб./мес.

Причины перезапуска на вебе:
- Сложное внедрение (высокий порог входа для администраторов)
- Дорогая разработка и поддержка на 1С
- Кривая воронка продаж (дубли, некорректные отчёты)
- Невозможность адаптировать под пакетные и разовые услуги

### 1.3. Целевые сроки

| Веха | Дата | Что |
|---|---|---|
| MVP | 1 июня 2026 | Переезд 20 текущих клиентов (календарный абонемент) |
| Обкатка | Июнь-Июль 2026 | Обратная связь, дорихтовка |
| Школы | Август 2026 | Фикс-абонемент для частных школ/садов |
| Расширение | Ноябрь 2026 | Пакетные абонементы (фитнес-студии), разовые услуги |

### 1.4. Критерий успеха

500 000 руб./мес. чистой прибыли.

## 2. Глоссарий

> Единые термины для всей команды. Согласовано устно 10.02.2026.

| Термин | Определение | Старый термин в 1С |
|---|---|---|
| **Лид** | Контакт на этапе воронки продаж (до первой оплаты абонемента). НЕ является клиентом. | Лид, Контрагент |
| **Клиент** | Физическое лицо, которое платит за услуги. Становится клиентом после первой оплаты. В детском центре — родитель. В фитнес-студии — сам посетитель. Центральная сущность системы. | Контрагент, Плательщик |
| **Подопечный** | Тот, кто получает услугу (если отличается от клиента). В детском центре — ребёнок. Информационная сущность — не участвует в финансовой логике. | Ученик, Ребёнок |
| **Абонемент** | Предоплата за определённое количество определённых услуг. Привязан к клиенту. Содержит информацию о подопечном, направлении, группе. Каждый месяц — новый абонемент (важно для LTV). | Договор, Направление, Интерес лида |
| **Направление** | Тип услуги: "Английский", "Скорочтение", "Стрейчинг", "Массаж". Определяет базовую стоимость занятия и длительность (например, скорочтение — 1 час, подготовка к школе — 1,5–2,5 часа). | Направление деятельности |
| **Группа** | Конкретный слот в расписании: направление + кабинет + время + инструктор + состав учеников. | Группа обучения |
| **Занятие** | Один конкретный урок/сеанс в расписании. Имеет дату, время, группу, инструктора. | Занятие |
| **Посещение** | Факт присутствия или отсутствия клиента/подопечного на занятии. Имеет тип (вид дня). | Посещение, Вид дня |
| **Списание** | Снятие суммы с баланса абонемента клиента за одно занятие (при явке или прогуле). | Списание |
| **Перерасчёт** | Занятие без списания (уважительная причина, справка). Потеря выручки для центра. | Перерасчёт, Неявка уважительная |
| **Отработка** | Занятие без списания, но с пометкой, что ребёнку нужно допзанятие. | — (новое) |
| **Филиал** | Физический адрес/локация бизнеса. | Подразделение, Структурная единица |
| **Кабинет** | Помещение внутри филиала, где проводятся занятия. | Кабинет |
| **Инструктор** | Сотрудник, проводящий занятия. Имеет ставку. Термин «педагог» не используется по юридическим причинам (лицензирование образовательной деятельности). | Преподаватель, Педагог |
| **Оборот** | Сумма всех поступлений денежных средств за период (ДДС). Когда деньги пришли. | Приход |
| **Выручка** | Сумма всех отработанных (списанных) занятий за период. Когда услуга оказана. | Отработанные абонементы, Доход |
| **Маржа** | Выручка минус переменные расходы (ЗП инструкторов + материалы). Ключевой показатель для расчёта ТБУ. | — |
| **Рентабельность** | Чистая прибыль делённая на выручку, в процентах. | — |
| **Финрез** | Финансовый результат: Выручка − Расходы = Прибыль/Убыток. | Финансовый результат |
| **Партнёр** | Организация (юрлицо/ИП), которая покупает подписку на нашу CRM-систему. НЕ путать с «клиентом» — клиент существует внутри системы партнёра. | — (новое, уровень SaaS) |

## 3. Карта сущностей

### 3.1. Диаграмма связей

```
Организация (tenant)
├── Филиал (1..N)
│   ├── Кабинет (1..N)
│   ├── Касса (1..N)
│   └── Банковский счёт (0..N)
│
├── Направление (1..N)
│   └── Базовая стоимость занятия, стоимость пробного, длительность занятия (мин)
│
├── Сотрудник (1..N)
│   ├── Роль (руководитель, управляющий, админ, инструктор, бухгалтер, только чтение)
│   ├── Личное дело: ФИО, контакты, ДР, дата начала работы
│   ├── Документы (загрузка PDF: паспорт, трудовой договор, СНИЛС и т.д.)
│   ├── История выплат ЗП
│   ├── Список групп (для инструкторов)
│   └── Ставка ЗП (0..N) — привязана к сотруднику + филиалу + направлению (любая комбинация; окладная ЗП может быть без привязки к филиалу/направлению)
│
├── Группа (1..N)
│   ├── → Филиал
│   ├── → Кабинет
│   ├── → Направление
│   ├── → Инструктор (может меняться)
│   ├── Шаблон расписания (дни + время для каждого дня + длительность)
│   ├── Лимит учеников
│   └── Занятие (генерируется из шаблона)
│       └── Посещение (по каждому ученику)
│           ├── Тип (вид дня)
│           ├── Сумма списания
│           └── Начисление ЗП инструктору
│
├── Лид / Клиент (1..N)
│   ├── ФИО (опционально), телефон (опционально), соцсети (ссылка VK/Telegram/другое, опционально)
│   │   └── Обязательно хотя бы одно из: телефон ИЛИ соцсети
│   ├── Доп. телефон, email, канал привлечения (заполняется позже, не при первом контакте)
│   ├── История комментариев (лента заметок: кто, когда, что)
│   ├── Подопечный (0..N) — информационные поля
│   │   └── Имя, возраст, ДР, доп. поля (произвольные)
│   ├── Абонемент (0..N)
│   │   ├── → Направление
│   │   ├── → Группа
│   │   ├── → Подопечный (опционально)
│   │   ├── Тип (календарный / фикс / пакетный)
│   │   ├── Баланс (сумма на счёте)
│   │   ├── Стоимость занятия (может быть индивидуальной)
│   │   ├── Скидки (0..N)
│   │   └── Статус (активный / закрыт)
│   ├── Воронка лидов (для новых контактов)
│   │   ├── Новый лид
│   │   ├── Назначено пробное
│   │   ├── Посетил пробное
│   │   ├── Ожидание оплаты
│   │   ├── Активный клиент ← (покидает воронку)
│   │   ├── Потенциал (отложенный)
│   │   ├── Нецелевой
│   │   ├── Чёрный список (только руководитель/управляющий, с причиной)
│   │   └── Архив
│   ├── Работа с клиентской базой (для действующих/бывших)
│   │   ├── Активный
│   │   ├── Допродажа (второе направление)
│   │   ├── Выбывший → Возврат
│   │   └── Архив
│   ├── Баланс клиента (общий, остатки с закрытых абонементов)
│   ├── Платежи (0..N)
│   └── LTV (рассчитывается: по деньгам + по количеству абонементов)
│
├── Платёж (1..N)
│   ├── Дата
│   ├── Сумма
│   ├── Тип (нал / безнал)
│   ├── → Клиент
│   ├── → Абонемент (опционально)
│   ├── → Касса / Банковский счёт
│   └── Направление (приход / расход / возврат / перемещение)
│
├── Расход (1..N)
│   ├── Дата
│   ├── Сумма
│   ├── Статья расхода (ДДС)
│   ├── Тип (постоянный / переменный)
│   ├── → Филиал (один, несколько или все)
│   ├── → Кабинет (один, несколько или все)
│   ├── → Направление (одно, несколько или все)
│   ├── → Инструктор (опционально, для обучения/сертификации)
│   ├── Период амортизации (опционально)
│   └── → Касса / Банковский счёт
│
├── Скидка (шаблон)
│   ├── Название
│   ├── Тип (постоянная / разовая / связанная)
│   ├── Значение (сумма или %)
│   └── Условие связи (для связанных: второе направление, второй ребёнок)
│
├── Причина отчисления (справочник)
│   └── Название (нет финансов, некому водить, устал, неинтересно, вернёмся позже, другое)
│
├── Статья расхода (справочник)
│   ├── Название
│   └── Тип (постоянный / переменный)
│
└── Вид дня (справочник, настраиваемый)
    ├── Название, сокращение
    ├── Списание с клиента (да/нет)
    ├── Процент списания (0-100%)
    ├── Начисление ЗП инструктору (да/нет)
    └── Цвет
```

### 3.2. Ключевые правила связей

1. **Лид → Клиент**: лид становится клиентом после первой оплаты абонемента
2. **Клиент → Абонементы**: один клиент может иметь N абонементов (разные направления, разные подопечные). Каждый месяц — новый абонемент
3. **Абонемент → Группа**: один абонемент привязан к одной группе (или нескольким для пакетного типа в v2)
4. **Группа → Инструктор**: в каждый момент времени один инструктор, но может меняться (замена). ЗП замены — по ставке заменяющего
5. **Занятие → Посещения**: одно занятие порождает N посещений (по числу учеников в группе)
6. **Посещение → Списание + ЗП**: каждое посещение (в зависимости от вида дня) может создать списание с абонемента и начисление инструктору
7. **Расход → Распределение**: постоянные расходы автоматически распределяются пропорционально выручке по направлениям/филиалам. Расход может быть привязан к нескольким (не только одному или всем) филиалам
8. **Воронка продаж + работа с клиентской базой**: воронка лидов (новые контакты, с маркетинговой атрибуцией) и работа с клиентской базой (допродажи, возвраты — без маркетинговой атрибуции, это не воронка продаж, а операционная работа с лояльностью)

## 4. Роли пользователей

### 4.1. Руководитель / Собственник

- Полный доступ ко всему
- Настройка системы (филиалы, кабинеты, направления, цены, ставки)
- Финансовые отчёты (P&L, маржа, прогноз прибыли, ДДС, ТБУ)
- Управление сотрудниками и правами доступа
- Контроль скидок и корректировок
- Внесение в чёрный список (с обязательной причиной)
- Закрытие и переоткрытие периодов
- Настройка прав управляющего

### 4.2. Управляющий (новая роль)

- Права **настраиваются руководителем** индивидуально
- Обычно: доступ к финансам, отчётам, управлению сотрудниками — но руководитель может скрыть финрез и другие данные
- Внесение в чёрный список
- Позиция между руководителем и администратором

### 4.3. Администратор (= Менеджер)

> Администратор и менеджер — одна роль с одинаковыми полномочиями.

- Работа с воронкой продаж (лиды, пробники, запись)
- Приём оплат
- Внесение расходов (простая форма: сумма + категория + филиал)
- Просмотр должников
- Отметка посещений (если инструктор не отмечает)
- Просмотр расписания, загруженности
- **Доступ к клиентской информации**: карточки клиентов, ДР, контакты, история, архивные/выбывшие клиенты
- **Работа с допродажами** действующим клиентам
- **Отчёт пропусков**: видит пропуски → звонит клиентам
- **Доступ к филиалам**: настраиваемый (конкретный набор, не только один или все)
- **НЕ видит** (по умолчанию): зарплаты, финрез, настройки ставок. Руководитель может расширить доступ (например, разрешить выплату ЗП)

### 4.4. Инструктор

- Просмотр своего расписания и состава групп
- Отметка посещений в своих группах
- Просмотр своей зарплаты (настраиваемо руководителем)
- **Видит**: ФИ детей (подопечных), возраст, дату рождения
- **НЕ видит**: телефоны, email, Telegram и другие контактные данные клиентов; балансы абонементов; зарплаты других сотрудников; финансовые отчёты

### 4.5. Бухгалтер

- Разнесение расходов по статьям
- Выплата зарплат
- Закрытие финреза
- Просмотр ДДС, остатков, планов платежей
- **НЕ видит**: воронку продаж, CRM-данные

### 4.6. Только чтение (новая роль)

- Просмотр отчётов и дашбордов (настраиваемо)
- Для инвесторов, партнёров, внешних консультантов
- **НЕ может** редактировать данные

## 5. User Stories

### 5.1. Воронка продаж (CRM-слой)

#### US-001: Создание лида при входящем звонке
**Как** администратор,
**я хочу** быстро внести данные нового контакта при звонке,
**чтобы** не потерять контакт и начать продажу.

**Поля формы:**
- ФИО (родителя) — опционально (иногда не называют имя)
- Телефон — опционально (проверка на дубли и чёрный список при заполнении)
- Соцсети (ссылка VK / Telegram / другое) — опционально
- **Обязательно хотя бы одно из**: телефон ИЛИ соцсети (минимум один способ связи)
- Канал привлечения (откуда узнали) — опционально (заполняется при последующих контактах)
- Желаемое направление — опционально
- Желаемый филиал — опционально
- Имя подопечного (ребёнка) — опционально
- Возраст подопечного — опционально
- Комментарий — опционально

**Минимальный сценарий:** Администратор создаёт лида только с номером телефона (или только со ссылкой на ВК). Остальные поля заполняются позже, когда лид приходит и заполняет анкету.

**Результат:** Лид создан в статусе "Новый лид". Появляется в воронке продаж.

**Защита:** При совпадении телефона — предупреждение. Если номер в чёрном списке — блокировка с отображением причины.

---

#### US-002: Назначение пробного занятия
**Как** администратор,
**я хочу** записать лида на пробное занятие в конкретную группу,
**чтобы** он пришёл, попробовал и купил абонемент.

**Действия:**
1. Открыть карточку лида
2. Выбрать группу (система показывает свободные места)
3. Выбрать дату пробного (из расписания группы)
4. Подтвердить запись

**Результат:** Статус лида → "Назначено пробное". Дата пробного видна в карточке и в общем списке. Инструктор видит пробника в составе группы на эту дату.

**Примечание:** Лид может иметь несколько пробных на разных направлениях (каждое — отдельная «заявка»).

**Автозадача:** За 24 часа до пробного → задача "Отправить напоминание [ФИО]".

---

#### US-003: Фиксация результата пробного
**Как** администратор,
**я хочу** отметить, что клиент пришёл/не пришёл на пробное,
**чтобы** двигать его по воронке и считать конверсию.

**Варианты:**
- **Пришёл** → статус "Посетил пробное"
- **Не пришёл** → статус остаётся, создаётся задача "Перезвонить [ФИО] — не пришёл на пробное"
- **Отменил** → статус "Потенциал" с комментарием и датой следующего контакта

**Оплата пробного:**
- Если пробное **бесплатное** (настройка по направлению) — ничего не списывается, инструктору не начисляется
- Если пробное **платное** — стоимость пробного из настроек направления. Инструктору начисляется ЗП
- Если после платного пробного клиент купил абонемент — стоимость пробного засчитывается (не берётся повторно)

#### US-004: Продажа абонемента
**Как** администратор,
**я хочу** оформить покупку абонемента после пробного,
**чтобы** клиент стал активным и попал в расписание.

**Действия:**
1. Открыть карточку лида (статус "Посетил пробное")
2. Нажать "Создать абонемент"
3. Выбрать: направление, группа, подопечный (если есть)
4. Система показывает: стоимость занятия, кол-во занятий до конца месяца, сумму абонемента
5. Применить скидку (опционально)
6. Подтвердить

**Результат:** Абонемент создан. Статус → "Ожидание оплаты". Подопечный добавлен в группу. Занятия в расписании отмечены как "Расписание" (не отмечен).

#### US-005: Приём оплаты
**Как** администратор,
**я хочу** внести оплату от клиента,
**чтобы** баланс абонемента обновился и клиент стал активным.

**Действия:**
1. Открыть карточку лида/клиента
2. Нажать "Принять оплату"
3. Ввести сумму
4. Выбрать способ (нал / безнал)
5. Выбрать кассу / счёт
6. Привязать к абонементу (если у клиента несколько)

**Результат:** Баланс абонемента увеличен. Если это первая оплата по абонементу — **автоматически**: статус → "Активный клиент", лид покидает воронку, подопечный появляется в расписании группы. Ручного перевода в "Активный" не требуется — всё происходит при факте оплаты.

**Примечание:** Клиент может переплатить (остаток переносится). Может недоплатить (долг фиксируется, технических ограничений нет).

#### US-006: Работа с воронкой (ежедневный ритуал админа)
**Как** администратор,
**я хочу** утром видеть список: кому позвонить, кто идёт на пробное, от кого ждём оплату,
**чтобы** никого не потерять.

**Дашборд "Задачи на сегодня":**
- Лиды с датой связи = сегодня → "Позвонить"
- Пробные назначены на **завтра** → "Отправить напоминание"
- Пробные назначены на сегодня → "Подтвердить пробное"
- Пробные вчера/позавчера без результата → "Узнать результат пробного"
- Статус "Ожидание оплаты" > 3 дней → "Напомнить об оплате"
- Дни рождения подопечных сегодня → "Поздравить"
- **Пропуски за вчера** → "Уточнить причину" (утренний ритуал: кто вчера не пришёл?)

**Сортировка лидов:** список автоматически сортируется по дате следующего контакта (сегодняшние сверху, отложенные на лето — внизу). При приближении даты контакта лид «всплывает» наверх. Это заменяет отдельную функцию предзаписи.

Задачи генерируются **автоматически** из дат в карточках.

#### US-007: Управление статусами лида
**Как** администратор,
**я хочу** перевести лида в "потенциал", "нецелевой", "чёрный список" или "архив",
**чтобы** он не засорял рабочий список.

**Статусы воронки лидов:**
| Статус | Описание | Условие перехода | Кто может |
|---|---|---|---|
| Новый лид | Первичный контакт | Создание карточки | Админ |
| Назначено пробное | Записан на пробное | Администратор назначил дату | Админ |
| Посетил пробное | Был на пробном | Отметка посещения | Админ |
| Ожидание оплаты | Согласился купить | Администратор перевёл | Админ |
| **→ Активный клиент** | Оплатил абонемент | Первая оплата | Автоматически |
| Потенциал | "Позвоните позже" | Администратор + дата связи | Админ |
| Нецелевой | Мусорный контакт | Администратор | Админ |
| Чёрный список | Запрет на контакт, обязательна причина | Только руководитель/управляющий | Руководитель / Управляющий |
| Архив | Не дозвонились, потеряли контакт | Администратор (без автоперевода) | Админ |

#### US-008: Допродажа и возврат клиента (работа с клиентской базой)
**Как** администратор,
**я хочу** оформить действующему клиенту второе направление или вернуть выбывшего,
**чтобы** наращивать LTV без дублирования в воронке лидов.

**Сценарии:**
- **Допродажа**: действующий клиент хочет второе направление → назначаем пробное (опционально) → создаём абонемент. Не попадает в воронку лидов, не учитывается в маркетинговой аналитике
- **Возврат**: выбывший клиент хочет вернуться → новый абонемент (не восстановление старого). Может пройти пробное или сразу купить. Не попадает в воронку лидов

**Результат:** Отдельная аналитика по допродажам и возвратам. Маркетинговые расходы не размываются. Допродажи не попадают в маркетинговую аналитику, **но** учитываются в бонусе администратора (v1.1).

### 5.2. Операционный слой

#### US-010: Отметка посещений
**Как** инструктор (или администратор),
**я хочу** отметить присутствие/отсутствие каждого ученика на занятии,
**чтобы** автоматически рассчитались списания и моя зарплата.

**Действия:**
1. Открыть расписание → выбрать занятие (или перейти из дашборда "Занятия сегодня")
2. Видеть список учеников группы
3. Кнопка **"Отметить всех"** — проставляет "Явка" всем ученикам сразу. Далее убрать галочки у отсутствующих и выбрать им нужный вид дня
4. Или по каждому ученику вручную выбрать вид дня: Явка / Прогул / Перерасчёт / Отработка / другой настроенный вид
5. Сохранить

**Автоматически:**
- По каждому ученику с видом дня, где "списание = да" → создаётся списание с абонемента (стоимость занятия × % списания)
- По каждому ученику с видом дня, где "ЗП = да" → создаётся начисление инструктору по его ставке
- Неотмеченные ученики остаются в статусе "Расписание" (прогноз)
- "Отработка" — не списывает, но помечает что ученику нужно допзанятие

**Отметка задним числом:** разрешена в пределах 10 дней (до закрытия периода).

#### US-011: Просмотр расписания
**Как** любой пользователь,
**я хочу** видеть расписание в удобном виде,
**чтобы** понимать, кто где когда занимается.

**Основной вид — неделя** (7 дней на экране, расписание повторяется еженедельно).

**Виды:**
- **По кабинетам** (сетка: строки = кабинеты, столбцы = время) — вид "день"
- **По инструкторам** (сетка: строки = инструкторы, столбцы = время) — вид "день"
- **По группам** (список групп с расписанием на неделю)
- **По направлениям** (фильтр: показать только "Английский" по всем филиалам/кабинетам)

**Фильтры:** филиал, направление, инструктор, кабинет — комбинируемые.

**Цветовая индикация заполняемости:**
- Зелёный: 100% мест занято
- Жёлтый: 51-99%
- Голубой: < 50%
- Серый: нет учеников

#### US-012: Создание и настройка группы
**Как** руководитель,
**я хочу** создать группу с расписанием,
**чтобы** в неё можно было записывать учеников.

**Поля:**
- Название (подробное: "Англ. 3 класс пн/ср 15:00")
- Направление
- Филиал
- Кабинет
- Инструктор
- Лимит учеников
- Шаблон расписания: дни недели + время начала для каждого дня (может различаться) + длительность (мин)
- Срок действия (бессрочная или с датами начала/конца)

**Результат:** Из шаблона автоматически генерируются занятия в расписание на указанный период.

**Генерация расписания:**
- На следующий месяц (автопродление из шаблонов)
- На произвольный период (например, на 9 месяцев / учебный год) — система спрашивает «на сколько месяцев пролить расписание?»
- ИЛИ создание с нуля (август → сентябрь, новый учебный год)
- После генерации расписания — генерация абонементов
- Возможность корректировки: удалить/закрыть отдельную группу из сгенерированного расписания

**Закрытие группы:** Группу можно закрыть/архивировать (например, при переходе на новый учебный год). Закрытая группа не участвует в генерации расписания.

---

#### US-013: Замена инструктора в группе
**Как** руководитель или администратор,
**я хочу** заменить инструктора в группе (разово или постоянно),
**чтобы** зарплата начислялась корректно новому инструктору.

**Варианты:**
- **Разовая замена**: на конкретное занятие. ЗП за это занятие → заменяющему **по его ставке**.
- **Постоянная замена**: с определённой даты. Все будущие занятия → новому инструктору.

**Частота:** 1-2 раза в месяц у каждой организации.

---

#### US-014: Создание абонемента (календарный тип)
**Как** администратор,
**я хочу** создать клиенту абонемент на месяц,
**чтобы** система рассчитала стоимость и отслеживала посещения.

**Логика календарного абонемента:**
1. Берётся базовая стоимость занятия по направлению
2. Считается количество занятий в группе до конца месяца (из расписания, с учётом даты начала)
3. Стоимость абонемента = стоимость × кол-во занятий
4. Применяются скидки (если есть)
5. Каждое посещённое занятие → списание стоимости одного занятия

**Пример:** Английский, пн/ср, клиент пришёл 15 марта, осталось 5 занятий, 500 руб./занятие → абонемент = 2 500 руб.

**Каждый месяц — новый абонемент.** Абонемент создаётся после генерации расписания на следующий месяц. Старый абонемент закрывается, остаток переносится.

#### US-015: Закрытие периода
**Как** руководитель,
**я хочу** закрыть учётный период (месяц),
**чтобы** зафиксировать финансовые результаты и защитить данные от случайных изменений.

**Действия:**
1. Проверить, что все посещения отмечены (отчёт о пропусках)
2. Нажать "Закрыть период"
3. После закрытия: отметки посещений и финансовые операции за этот месяц блокируются

**Переоткрытие:** Руководитель может переоткрыть период (например, поступила справка задним числом), внести изменения и перезакрыть.

### 5.3. Финансовый слой

#### US-020: Внесение расхода (простая форма для админа)
**Как** администратор,
**я хочу** быстро внести расход (купили шоколадки для отзывов),
**чтобы** бухгалтерия была в курсе.

**Форма (максимально простая):**
1. Сумма — ввести число
2. Статья расхода — выбрать из списка (хозтовары, канцелярия, маркетинг...)
3. Филиал — выбрать (один, несколько или "все")
4. Оплачено из — выбрать кассу / счёт
5. Распределить на несколько месяцев? — опционально (кнопка «Распределить на N месяцев»). Пример: 12 000 руб. / 12 мес. = 1 000 руб./мес. в финрезе. В ДДС проходит полная сумма сразу
6. Комментарий — опционально

**Больше ничего.** Постоянный/переменный, направление — это настраивает руководитель/бухгалтер потом (или прописано в настройках статьи).

#### US-021: Просмотр должников
**Как** администратор,
**я хочу** видеть, кто не заплатил,
**чтобы** звонить и напоминать.

**Таблица должников (порядок столбцов):**
- Филиал
- Направление
- Клиент (ФИО)
- Абонемент (подопечный)
- Плановый долг (если посетит все занятия)
- Фактический долг (по уже отмеченным)
- Дата последней оплаты
- **Обещанная дата оплаты** (отдельный столбец, на неё ориентируемся)
- Комментарий

**Подсветка:** просроченная обещанная дата — красным. Не хватает на следующее занятие — предупреждение.

**Фильтры:** по филиалу, направлению, статусу (актив / отчислен), сумме долга.

**Автозадача:** Если обещанная дата оплаты = сегодня → задача "Позвонить [ФИО] — обещал оплатить".

#### US-022: Финансовый результат (месяц)
**Как** руководитель / бухгалтер,
**я хочу** увидеть финансовый результат за месяц,
**чтобы** понять прибыльность бизнеса.

> **Важно:** ФинРез = Выручка − Расходы **текущего периода**. Если расход распределён на несколько месяцев (см. US-020), в финрезе учитывается только доля текущего месяца. В ДДС — полная сумма в момент оплаты.

**Два формата отчёта:**

**Формат A — общий P&L (для собственника):**
```
Выручка (отработанные абонементы):     500 000
─────────────────────────────────────────────
Переменные расходы:
  Зарплата инструкторов:              -150 000
  Материалы:                          -20 000
─────────────────────────────────────────────
Маржа:                                 330 000 (66%)
─────────────────────────────────────────────
Постоянные расходы:
  Аренда:                             -80 000
  Интернет, связь:                     -5 000
  Канцелярия:                         -10 000
  Маркетинг:                          -30 000
  Прочее:                             -15 000
─────────────────────────────────────────────
Итого расходы:                       -310 000
─────────────────────────────────────────────
Чистая прибыль:                       190 000
Рентабельность:                           38%
```

**Формат B — по направлениям/филиалам (для управленческих решений):**
```
                   Англ.  Скорочт.  Развивайка  Итого
Выручка:           200к    180к       120к       500к
ЗП инструктора:    -60к    -50к       -40к      -150к
Расходы (доля):    -56к    -50к       -34к      -140к
─────────────────────────────────────────────────────
Прибыль:            84к     80к        46к       210к
```

Постоянные расходы (аренда, интернет и т.д.) распределяются пропорционально выручке направления.

#### US-023: Прогноз зарплаты инструкторов
**Как** руководитель,
**я хочу** видеть, сколько начислено и сколько осталось выплатить каждому инструктору,
**чтобы** планировать выплаты.

**Таблица:**
- Инструктор
- Начислено за месяц (по отмеченным занятиям)
- Премии
- Штрафы
- Выплачено
- Осталось выплатить = Начислено + Премии − Штрафы − Выплачено

**Детализация:** клик по сумме → список занятий с расшифровкой (дата, группа, кол-во учеников, ставка, сумма).

#### US-024: Выплата зарплаты
**Как** руководитель / бухгалтер,
**я хочу** провести выплату сотруднику,
**чтобы** система знала остаток к выплате.

**Действия:**
1. Выбрать сотрудника
2. Ввести сумму выплаты (полная, частичная/аванс — произвольная сумма)

4. Подтвердить

**Результат:** Создаётся расход (статья "Зарплата"), привязанный к сотруднику, автоматически относится на его филиал/направления.

**Периодичность:** произвольная (раз в месяц, дважды, по запросу).

#### US-025: Возврат средств клиенту
**Как** администратор / руководитель,
**я хочу** оформить возврат денег клиенту,

- При наличной оплате — возврат наличными

### 5.4. Настройки и администрирование

#### US-030: Первичная настройка организации
**Как** новый клиент системы,
**я хочу** быстро настроить систему под свой центр,
**чтобы** начать работать.

**Пошаговый мастер (wizard):**

| Шаг | Что |
|---|---|
| 1 | Создать филиал(ы): название, адрес, часы работы |
| 2 | Создать кабинеты: название, привязка к филиалу |
| 3 | Создать направления: название, стоимость занятия, длительность, стоимость пробного (платное/бесплатное) |

| 7 | Создать кассы / банковские счета |
| 8 | Настроить статьи расходов (предзаполнены, можно добавить), постоянные расходы |
| 9 | Настроить каналы привлечения (предзаполнены, можно добавить) |
| 10 | Настроить виды скидок (связанные, несвязанные, %, сумма) |
| 11 | Создать логины и пароли сотрудникам |

**Целевое время:** требует уточнения (см. открытый вопрос #19). Предыдущая оценка 45 мин. признана нереалистичной.
Текущее время в 1С: несколько часов + сопровождение специалиста (4+ встречи).

**Причина сложности 1С:** не сложность данных, а сложность навигации. Кнопки названы непонятно, приходится «проваливаться» в глубокие подменю.

## 7. Приоритизация

### 7.1. MVP (к 1 июня 2026)

**Цель:** переехать 20 текущих клиентов с календарным абонементом.

**Модули:**
- CRM: воронка лидов + работа с клиентской базой, полный цикл (лид → активный клиент → допродажа/возврат). Обзвон (блок кампаний). История коммуникации. Детализация оттока
- Справочники: филиалы, кабинеты, направления, сотрудники, виды дней, причины отчисления
- Расписание: группы, занятия, отметка посещений, закрытие периода. Разовые занятия/отработки
- Абонементы: календарный тип, скидки (3 типа), баланс, возврат
- Финансы: оплаты, расходы (простая форма), должники, ДДС, остатки, прогноз прибыли, финрез (оба формата с маржой и рентабельностью), перемещение между кассами. **Drill-down во всех отчётах**. Сверка кассы
- Зарплата инструкторов: ставки (3 формата, включая на группу), начисления, выплаты, премии/штрафы, карточка сотрудника. Корректировки прошлых периодов
- Дашборд: настраиваемый по ролям главный экран, задачи (ручные + автоматические + привязанные к датам), уведомления
- Администрирование: мультитенант, 6 ролей, wizard, импорт, аудит

**Всего требований MVP:** ~95

## 8. План миграции

### 8.1. Что переносим

| Данные | Источник | Формат | Обработка |
|---|---|---|---|
| Клиенты (родители) | Выгрузка из 1С | CSV/Excel | Нормализация: дедупликация по телефону, очистка ФИО, структурирование (родитель → подопечные) |
| Остатки балансов | Выгрузка из 1С | CSV | Загрузка в абонементы: долг/переплата по каждому клиенту-направлению |
| Остатки денег | Ручной ввод | — | Начальные остатки касс и счетов |

### 8.2. Что НЕ переносим (вводится заново)

- Инструкторы и сотрудники (< 20, руками)
- Расписание (создаётся заново — удобный момент: начало учебного года)
- История оплат и посещений (не нужна для работы)
- Воронка продаж / конверсии (начинаем с чистого листа)
- Склад (при реализации — ревизия + ручной ввод)

### 8.3. Процесс переезда

| Этап | Когда | Что |
|---|---|---|
| 1. Предупреждение | 1 мая | Письмо клиентам: "через месяц переезжаем, новеньких заводите в новую систему" |
| 2. Настройка | 1-31 мая | Клиенты настраивают справочники в новой системе (wizard + видеоинструкция) |
| 3. Импорт | 1-7 июня | Загрузка клиентских баз + остатков |
| 4. Параллельная работа | 1-14 июня | Две недели на адаптацию |
| 5. Отключение 1С | 15 июня | Полный переход |
| 6. Сбор обратной связи | Июнь-Июль | Доработки по реальному использованию |

**Выгрузку из 1С делаем мы** (единая версия у всех 20 клиентов).

### 8.4. Логика нормализации при импорте

1. **Дедупликация:** группировка по номеру телефона → все записи с одним телефоном = один клиент
2. **Клиент/Подопечный:** в 1С запись "Иванов Петя, мама Иванова Мария" → Клиент: Иванова Мария (тел.), Подопечный: Петя
3. **Остатки:** сумма баланса из 1С → начальный баланс абонемента в новой системе. Если долг — отрицательный баланс. Если переплата — положительный
4. **Валидация:** отчёт для клиента: "Вот ваши данные после импорта, проверьте"
