## 1. Модель: три инструмента

| Инструмент | Как применяется | К чему |
|---|---|---|
| **Тип 1 «Скидка за второй абонемент»** | автоматически, всегда | календарные абонементы в рамках одного месяца периода |
| **Тип 2 «Постоянная скидка»** (шаблоны) | вручную, в карточке родителя | все новые абонементы родителя, включая пакетные |
| **Разовая скидка** (бонус на баланс) | вручную, кнопка в карточке | баланс родителя; без изменений (как сейчас) |

Правила сочетания:
- Скидки **не суммируются**: на одном абонементе действует ровно одна.
- **Тип 2 приоритетен и эксклюзивен**: если у родителя выбран шаблон типа 2 — тип 1 для него не действует вообще, даже если тип 1 выгоднее.
- Вручную можно выставить только тип 2 либо сделать разовую. Тип 1 на уровне клиента не выбирается и **не отключается** — только глобальный тоггл в настройках шаблонов организации.

## 2. Тип 1 — «Скидка за второй абонемент»

Системный шаблон, один на организацию (объединяет старые «За 2-го ребёнка» и «За 2-е направление»). Настраивается: вкл/выкл (`isActive`), тип значения (процент | фикс-скидка за занятие), размер.

**Условие:** у родителя в одном календарном месяце периода (`periodYear`+`periodMonth`) больше одного живого календарного абонемента — независимо от количества детей и направлений. Второй абонемент того же ребёнка в том же направлении в том же месяце тоже считается.

**Получатели:** все абонементы месяца, **кроме самого дорогого**. «Самый дорогой» — по `totalAmount` (стоимость БЕЗ скидок). При равенстве стоимости освобождается выписанный **раньше** (порядок: `createdAt`, затем `id` — детерминированно и внутри пачки массовой выписки), скидка — на выписанные позднее.

**Состав месяца:**
- **Штатно закрытый** (отходивший своё) абонемент **остаётся в составе месяца** — он состоялся и продолжает «прикрывать» скидки остальных. Иначе закрытие дорогого абонемента в 20-х числах (расписания детей заканчиваются в разные дни) снимало бы скидки с долгами на ровном месте.
- **Отчисленный и удалённый** — выпадают из состава (абонемент аннулирован), оставшиеся пересчитываются (§3, §5).

**Не участвуют:**
- пакетные абонементы (нет месяца периода; к ним применим только тип 2);
- абонементы прошедших месяцев: при выписке абонемента на уже прошедший месяц автоматика скидок не срабатывает вовсе («второй июньский, выписанный 2 июля» скидку не получает и пересчёт не запускает). При необходимости корректировка — вручную **разовой скидкой** (бонус на баланс).

Пример (из решения): 25 июня при живых июньских абонементах выписывается первый абонемент на июль → скидки нет, он первый В ИЮЛЕ. Появится второй июльский — скидка применится к более дешёвому из июльских.

## 3. Скидка живёт в цене занятия

Ключевой принцип: скидка — это **снижение цены каждого занятия**, а не вычет из итоговой суммы.

- Эффективная цена занятия = `lessonPrice − скидка_за_занятие`. Процентная: `lessonPrice × (1 − p/100)`. **Ограничение: не ниже 0** (скидка ≥ цены → занятие бесплатно; процент капится 100%).
- **Списания (Attendance.chargeAmount) идут по эффективной цене.** Уже сделанные списания не пересчитываются никогда.
- Применение/снятие/замена скидки на абонементе с отработанными занятиями действует **только на оставшиеся занятия** (пропорция по занятиям, не по дням): скидка отменена после 2 из 8 отхоженных — клиент сохраняет скидку за 2 прошедших занятия, остальные 6 — по полной цене (снимается 6/8 скидки).
- **«Отхожено» считается по числу отметок** (Attendance по абонементу, факт, без назначенных отработок и pending), НЕ по `chargeAmount > 0` — иначе бесплатные занятия (100% скидка) не считались бы отхоженными. Остаток = `max(0, totalLessons − отхожено)`. Существующие места подсчёта по `chargeAmount > 0` (закрытие, отчёт остатков, активные абонементы) мигрируются.
- **Выручка** (= сумма списаний) уменьшается на скидку. **ЗП инструктора «% от оплат»** автоматически следует за ценой со скидкой (расчёт уже идёт от `chargeAmount` — правок не требует, нужен тест). Схемы «за ученика», «за занятие», «фикс за выход» от цены не зависят.
- Стоимость абонемента (`finalAmount`) = снимок прошлых списаний + остаток × эффективная цена. `discountAmount` — производная (`totalAmount − finalAmount`).

Это решение попутно закрывает существующий баг: сейчас списания идут по полной цене, и закрытие полностью отходившего абонемента со скидкой вешает на клиента ложный долг в размере скидки (paid = finalAmount < charged = totalAmount).

## 4. Деньги при пересчёте

- **Универсальная формула возврата:** возврат = `max(0, оплачено − новая стоимость)` → на баланс родителя. Если оплачено ≤ новой стоимости — возврата нет, меняется долг (`balance`). Пример: оплачен 3000, появился второй абонемент 3200 → скидка 10% на меньший → возврат 300, абонемент стоит 2700.
- **Оформление возврата:** проводка на баланс родителя (новый тип `discount_refund`) + **сторно-платёж по абонементу**: `Payment { type: transfer_in, amount: −X }` на первый активный счёт (зеркало pay-from-balance). Так «оплачено» уменьшается, закрытие/отчисление не вернёт те же деньги второй раз, а касса/ДДС не затронуты (transfer_in в ДДС и остатки счетов не входит). НЕ через `/api/payments/refund` — тот выводит деньги со счёта. В истории баланса — строка «Возврат по скидке».
- **Удорожание** (скидка снята с оставшихся занятий): растёт долг по абонементу — только в части оставшихся занятий (§3). Оплаченный абонемент может стать должником — осознанно (пропорция по занятиям ограничивает сумму).
- **Бесплатный абонемент** (стоимость 0 после скидки): автоактивация — и при создании, и когда пересчёт опускает «к оплате» до нуля у pending. Активация повторяет побочные эффекты оплаты (activatedAt, заявка воронки → won, GroupEnrollment.paymentStatus, Ward.salesStage) — общий код с pay-from-balance.
- Корректировка оплаченных абонементов **разрешена** (запрет снят) — в рамках правил выше.

## 5. Мониторинг — триггеры пересчёта

Пересчёт выполняется по клиенту (тип 2 — целиком; тип 1 — по затронутым месяцам) при событиях:

1. **Создание абонемента** — все три пути: карточка клиента (`POST /api/subscriptions`), воронка продаж (`move-to-awaiting-payment` — сейчас скидки не применяет, добавить), массовая выписка (`bulk-renew` — **один пересчёт на клиента после всей пачки**).
2. **Отчисление / удаление** абонемента месяца → выбывший исключается из состава, оставшиеся пересчитываются (только оставшиеся занятия). Штатное закрытие пересчёт НЕ запускает (§2). Точки: PATCH status=withdrawn, `/refund`, `DELETE /api/subscriptions/[id]` (сейчас вообще без пересчёта), кроны `close-unpaid-subscriptions` (как отчисление) — судьба денег удалённого абонемента: §11.
3. **Изменение цены/числа занятий** абонемента (PATCH) — допущение, см. §11: без этого «самый дорогой по totalAmount» ломается при правках.
4. **Смена шаблона у клиента** (тип 2 поставлен/снят, PATCH `/api/clients/[id]`) — см. §6.
5. **Включение тоггла типа 1** — разовый пересчёт месяцев начиная со следующего (§7).

Пересчёт затрагивает только текущий и будущие месяцы (прошедшие — §2). Деактивация/перевод зачисления (enrollments) триггером НЕ являются — абонемент не меняется (вызовы старого `recalcLinkedDiscounts` оттуда удаляются вместе с ним).

## 6. Тип 2 — постоянные шаблоны

- Системный шаблон «Постоянная скидка» + любое число пользовательских («+ Шаблон»). Все работают одинаково (kind=permanent). Старые типы шаблонов больше не создаются.
- Выбор в карточке родителя: применяется ко всем **новым** абонементам родителя (без условий по количеству; пакетные включительно).
- При выборе типа 2 у клиента с действующими скидками типа 1: тип-1-скидки **заменяются на тип 2 с перерасчётом оставшихся занятий** (прошедшие списания не трогаем; доплата/возврат — §4). Существующие абонементы **без** скидки (включая «самый дорогой») не трогаются до следующей выписки.
- Снятие типа 2 («Без скидки»): симметрично — тип-2-скидки снимаются с оставшихся занятий, тип 1 сразу пересчитывает текущий/будущие месяцы (допущение — §11).
- Поле скидки в карточке: «Без скидки» → автоматически показывает тип 1 (когда применён) → вручную переключается на тип 2. Значение «тип 1» в поле — индикатор, руками не выбирается.

## 7. Включение / выключение шаблонов (настройки организации)

- **Выключение** (тоггл типа 1 или деактивация шаблона типа 2): уже выданные скидки доживают до конца своих абонементов; новые абонементы — без скидки. Никакого пересчёта в момент выключения.
- **Включение типа 1:** действует только на абонементы с периодом **со следующего месяца** после включения (хранится `activatedAt`). Абонементы текущего месяца не трогаются — даже при выписке нового абонемента текущего месяца после включения. В момент включения выполняется разовый пересчёт уже выписанных абонементов будущих месяцев (например, продлённых заранее) — допущение, см. §11.
- Изменение размера/типа значения шаблона: применяется к будущим применениям скидки; уже выданные не пересчитываются (допущение — §11).

## 8. Семантика фикс-скидки (UI и расчёт)

- `value` фикс-шаблона = **размер скидки за занятие** (раньше: «стоимость занятия со скидкой»). Подписи в форме шаблона и справке «?» обновить.
- Ограничения: фикс капится ценой занятия (бесплатно, не отрицательно); процент капится 100%.

## 11. Решения по допущениям (подтверждены владельцем 12.06.2026)

1. **Удаление абонемента с оплатами/списаниями — ЗАПРЕЩЕНО** (DELETE возвращает ошибку с подсказкой «отчислите/закройте»). Удаление остаётся только для ошибочно созданных абонементов без денег; такое удаление — триггер пересчёта месяца.
2. **Включение тоггла** делает разовый пересчёт уже выписанных абонементов будущих месяцев (продлённые заранее семьи получают скидку сразу) — да.
3. **Снятие типа 2** («Без скидки»): тип-2-скидки снимаются с оставшихся занятий, тип 1 сразу пересчитывает месяц — да.
4. **Изменение размера шаблона**: выданные скидки НЕ пересчитываются; UI при изменении размера явно предупреждает об этом факте.
5. **Редактирование цены/числа занятий абонемента** запускает пересчёт месяца — да.
6. **Тип 1 после миграции создаётся выключенным** (фикс 50 ₽/занятие) — организация включает сама.
7. **Откат отметки** возвращает занятие в «оставшиеся»; повторное списание — по текущей эффективной цене. Зафиксировать тестом.
