# Детализация ЗП преподавателя + частичная выплата по направлениям

**Дата:** 2026-06-22
**Раздел:** `/salary` (ЗП преподавателей)
**Тип:** доработка (UI + 1 новый API-эндпоинт), без изменения схемы Prisma

## Контекст и задача

Из запроса заказчика (бухгалтер):
1. На `/salary` нужно «проваливаться» в преподавателя по клику на ФИО и видеть, **за что конкретно** начислена ЗП.
2. Возможность выбрать **направления чекбоксами**, за которые сейчас выплачивается ЗП, и **изменить сумму к выплате** — для **частичной** выплаты. Это должно фиксироваться и корректно отражаться в итоговой выплате («Выплачено»/«Осталось»).

## Текущее состояние (что уже есть)

- `/salary/page.tsx` — таблица: *Сотрудник | Роль | Начислено | Премии | Штрафы | Выплачено | Осталось*, период через `MonthPicker`. ФИО — **не ссылка**, drill-down нет.
- Начисление: `Attendance.instructorPayAmount` (считается в `/api/lessons/[id]/attendance` через `resolveRate`+`calcPay`), привязано к занятию → группе → **направлению**.
- `GET /api/salary-payments/accruals` уже агрегирует по `(employee, directionId)` и отдаёт `byDirection[]` + employee-итоги (`accrued/bonuses/penalties/alreadyPaid/remaining`). Но `alreadyPaid`/`remaining` — **на уровне сотрудника**, не по направлениям; per-lesson детали не отдаёт.
- Модель выплат: `SalaryPayment` (шапка) + `SalaryPaymentItem(employeeId, accountId, **directionId**, amount, comment)`. **Уже поддерживает выплату частями по направлениям.**
- `POST /api/salary-payments` (режим `items[]`): создаёт `SalaryPayment` + `SalaryPaymentItem[]`, списывает счёт(а), пишет аудит, проверяет блокировку периода; доступ — **owner/manager**.
- Выплаты попадают в ДДС (`cash-flow` по `SalaryPayment.amount`) и в ОПИУ (по `SalaryPaymentItem.directionId`). Меняется `FinancialAccount.balance`.
- «Осталось» = `начислено + премии − штрафы − выплачено` (выплачено = Σ `SalaryPaymentItem.amount` за период).
- Корректировки: `SalaryAdjustment(type: bonus|penalty)` — **на уровне сотрудника, без направления**.

**Вывод:** факт частичной выплаты по направлениям уже фиксируется (`SalaryPaymentItem.directionId`). Нужны только: (1) экран детализации с drill-down, (2) поток частичной выплаты на нём, (3) per-direction остаток. **Новая Prisma-сущность не нужна** (отклонённая альтернатива «таблица сессий выбора» — YAGNI: выбор = какие `items` созданы).

## Дизайн

### 1. Страница детализации `/salary/instructor/[employeeId]?year=&month=`

- **Доступ:** owner/manager (полный, с выплатой); инструктор — только свою страницу (`session.employeeId === employeeId`), **read-only** (без чекбоксов/кнопки выплаты), как на главной `/salary`.
- **Server-shell** (`page.tsx`) + **клиентский компонент** (`instructor-salary-detail.tsx`), который тянет данные из нового GET-эндпоинта и держит состояние выбора.
- Период берётся из query (`year`,`month`), по умолчанию — текущий; вверху `MonthPicker`.

**UI (таблица направлений):**

```
Петров И.И. — июнь 2026                         Счёт: [Касса ▾]  Дата: [22.06.2026]
┌──┬─────────────┬──────────┬──────────┬───────────┬─────────────────┐
│☑ │ Направление │ Начислено│ Выплачено│  Остаток  │   К выплате      │
├──┼─────────────┼──────────┼──────────┼───────────┼─────────────────┤
│☑ │ ▾ Рисование │  12 400  │     0    │  12 400   │ [   12400   ]    │
│  │   05.06 Гр.«Малыш» «Был» 6 уч. — 720 ₽                          │
│  │   07.06 Гр.«Малыш» «Был» 5 уч. — 600 ₽                          │
│☐ │ ▸ Английский│   8 200  │     0    │   8 200   │ [       0   ]    │
│☑ │   Премии−штрафы (без направления)   +2 000  │   2 000   │ [ 2000 ] │
└──┴─────────────┴──────────┴──────────┴───────────┴─────────────────┘

```

- Строка направления **раскрывается** (▸/▾) в список занятий: дата, группа, вид дня, число оплачиваемых учеников, сумма ЗП.
- Отдельная строка **«Премии − штрафы»** (`directionId = null`) — net = премии − штрафы; чекбокс + поле суммы.
- Поле «К выплате» предзаполнено остатком; **редактируемое**. Если введено **больше остатка** — показываем предупреждение (аванс/переплата), но **разрешаем** (как в «Документе выплат»).
- «Итого к выплате» = Σ полей по **отмеченным** строкам.

### 2. Новый эндпоинт `GET /api/salary/instructor/[employeeId]?periodYear=&periodMonth=`

Доступ: `finance.salary` ИЛИ `session.employeeId === employeeId` (read-only).

Возвращает:
- `employee`: `{ id, name, role }`
- `byDirection[]`: `{ directionId: string|null, directionName, accrued, paid, remaining, lessonCount }`
  - `accrued` = Σ `Attendance.instructorPayAmount` по направлению за период (учёт `substituteInstructorId`, `instructorPayEnabled`), как в `accruals`.
  - `paid` = Σ `SalaryPaymentItem.amount` с этим `directionId` за период.
  - `remaining` = `accrued − paid`.
- `adjustments`: `{ bonuses, penalties, net, paidNoDirection, remaining }`
  - `paidNoDirection` = Σ `SalaryPaymentItem.amount` с `directionId = null` за период (сюда же попадают legacy/простые выплаты).
  - `remaining` = `net − paidNoDirection`.
- `lessons[]` (для раскрытия): `{ lessonId, date, groupName, directionId, directionName, typeName, studentsCharged, amount }` — агрегировано **по занятию** (Σ `instructorPayAmount` отметок этого преподавателя; `studentsCharged` = число оплачиваемых отметок).
- `totals`: `{ accrued, bonuses, penalties, paid, remaining }` (authoritative: `remaining = accrued + bonuses − penalties − paidTotal`).
- `accounts[]`: `{ id, name }` активных счетов (для селектора).
- `periodLocked`: boolean.

**Реконсиляция остатков** (документируем явно): Σ остатков по направлениям + остаток «Премии−штрафы» = итоговый остаток сотрудника, **если** все прошлые выплаты имели направление. Legacy/простые выплаты (`directionId=null`) уменьшают строку «Премии−штрафы» (`paidNoDirection`). Authoritative — `totals.remaining`.

Логика агрегации выносится в чистую функцию `buildInstructorSalaryDetail(...)` в `lib/salary/` для юнит-тестируемости.

### 3. Частичная выплата (на странице детализации)

- Кнопка **«Выплатить выбранное»** → `POST /api/salary-payments` (существующий, режим `items[]`):
  ```
  items = [
    { employeeId, accountId, directionId: <dir|null>, amount, comment }   // по каждой отмеченной строке с amount > 0
  ]
  ```
  - Направления → `directionId = <id>`; строка «Премии−штрафы» → `directionId = null`.
  - Дата/период — из выбранного периода/даты.
- После успешной выплаты — рефетч детализации; `paid`/`remaining` пересчитываются (поля «К выплате» сбрасываются на новый остаток).
- Серверная валидация (уже есть в POST): owner/manager, период не закрыт, счета/направления существуют. Доп. требований нет.

### 4. Главная `/salary`

- ФИО сотрудника → **ссылка** на `/salary/instructor/[id]?year=&month=` (период пробрасывается).
- В строке — кнопка/иконка **«Детали/выплата»** (дублирует ссылку для наглядности).
- Существующие диалоги «Провести выплату» и «Документ выплат» **не трогаем**. (Возможное удаление простого диалога — отдельной задачей, вне этого спека.)

### 5. PageHelp

- Новый ключ `salary/instructor/[employeeId]` (или `salary/instructor`) в `page-help-content.ts` + `<PageHelp>` на странице: объяснить детализацию, частичную выплату по направлениям, переплату-предупреждение, что «Премии−штрафы» платятся без направления.
- Обновить help `salary` (если есть): упомянуть переход в детализацию.

## Краевые случаи

- **Переплата по направлению** (введено > остатка): предупреждение, выплата разрешена; `remaining` может стать отрицательным — отображаем как есть.
- **Закрытый период**: кнопка выплаты заблокирована (`periodLocked`), POST вернёт 403.
- **Нет начислений**: страница показывает «Нет начислений за период».
- **Замена инструктора**: ЗП идёт замещающему (`substituteInstructorId`) — учитываем в агрегации (как в `accruals`).
- **Окладники** (`monthlySalary`): добавляются строкой по `defaultDirectionId` (как в `accruals`) — переиспользуем ту же логику.
- **Инструктор смотрит свою страницу**: read-only, без выплаты.

## Вне области (out of scope)

- Изменение ставок/схем начисления.
- Откат/удаление выплат и пересчёт при изменении посещений закрытого периода.
- Удаление старого простого диалога «Провести выплату».
- Изменения ДДС/ОПИУ (уже работают через `SalaryPaymentItem.directionId`).

## Затрагиваемые файлы

- **Новое:** `app/src/app/api/salary/instructor/[employeeId]/route.ts` (GET); `app/src/lib/salary/instructor-detail.ts` (чистая агрегация); `app/src/app/(dashboard)/salary/instructor/[employeeId]/page.tsx` (shell) + `instructor-salary-detail.tsx` (клиент).
- **Правка:** `app/src/app/(dashboard)/salary/page.tsx` (ФИО-ссылка + кнопка); `app/src/lib/page-help-content.ts` (+ ключ).
- **Тесты:** `app/src/__tests__/instructor-salary-detail.test.ts` — юнит на `buildInstructorSalaryDetail` (per-direction accrued/paid/remaining, adjustments, totals, реконсиляция с null-direction выплатой). По возможности — e2e happy-path (опционально, тяжёлый, на CI).

## Критерии готовности

1. Клик по ФИО на `/salary` открывает детализацию за тот же период.
2. Видны направления с начислено/выплачено/остаток и раскрытием в занятия; отдельная строка «Премии−штрафы».
3. Чекбоксы + редактируемые суммы; «Итого к выплате» считается по отмеченным; переплата — с предупреждением.
4. «Выплатить выбранное» создаёт `SalaryPaymentItem` по направлениям; «Выплачено»/«Осталось» на `/salary` и в детализации обновляются; запись видна в ДДС/ОПИУ.
5. Инструктор видит только свою детализацию, без выплаты.
6. tsc чисто, юнит зелёный, CI зелёный после деплоя.
