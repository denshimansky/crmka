# Отложенное отчисление абонемента (дата отчисления в будущем)

**Дата:** 2026-06-26
**Статус:** дизайн на согласовании
**Автор:** Денис + Claude

## Проблема

Сейчас дата отчисления абонемента не может быть в будущем (`validateWithdrawalDate`:
`«не может быть в будущем»`). Расчёт возврата/долга при отчислении делается по
**уже отмеченным** занятиям (`оплачено − стоимость отработанного`). Если бы дату
поставили вперёд, ребёнок остался бы в составе до этой даты, но деньги за ещё не
прошедшие занятия вернулись бы на баланс сразу — и эти занятия прошли бы
«бесплатно» (массовая отметка `chargeAmount=0`) или списались бы как разовые с
баланса (одиночная отметка). Нестыковка.

## Цель

Разрешить отчисление с датой в **будущем** (в пределах периода абонемента).
Ребёнок ходит до даты X включительно, **платит только за фактически посещённые**
занятия до X, остаток возвращается на баланс. Возврат — по фактическим пропускам
(если ребёнок перестал ходить раньше X, деньги за непосещённые занятия
возвращаются).

## Подход A — «Отложенное отчисление»

Отчисление с будущей датой = **запланировать уход на дату X**, а не закрыть
сейчас. Финальный денежный расчёт происходит на X по факту посещений
существующим механизмом списания/сверки. Никаких изменений в логике отметок.

### Поток

1. **Отметка даты в диалоге.** Дата ≤ сегодня → как сейчас (немедленное
   отчисление, поведение не меняется). Дата > сегодня → **запланированное**
   отчисление (новый путь).
2. **При планировании (дата X > сегодня):**
   - `Subscription.status` остаётся `active` (НЕ `withdrawn`); баланс/возврат НЕ
     трогаем.
   - Заполняем `scheduledWithdrawalDate=X`, `scheduledWithdrawalReasonId`,
     `scheduledWithdrawalComment`.
   - `GroupEnrollment.withdrawnAt = X+1` (как при обычном отчислении) — ребёнок в
     составе на занятиях ≤ X, после X выпадает из расписания.
   - Комментарий-заметка в историю коммуникаций («Запланировано отчисление на X»).
3. **До X:** ребёнок ходит, занятия ≤ X **списываются как обычно по факту**
   (абонемент `active`). Пропустил — не списалось. Никакого спец-кода.
4. **Финализация (cron на X+1):** для абонементов с
   `scheduledWithdrawalDate <= вчера` и `status IN (active, pending)`:
   - Та же сверка, что при ручном отчислении:
     `delta = нетто-оплачено − Σ Attendance.chargeAmount − прошлые сверки`.
   - `delta > 0` → возврат на баланс; `delta < 0` → долг на баланс; `balance=0`.
   - `status=withdrawn`, `withdrawalDate=scheduledWithdrawalDate`,
     `withdrawalReasonId=scheduledWithdrawalReasonId`; очистить `scheduled*`.
   - Пересчёт скидок; перевод клиента в «Выбывшие», если активных абонементов не
     осталось (переиспользуем `churnClientIfNoActiveSubscription`).
   - Зачисление в группе уже деактивировано на шаге 2 — не трогаем.

### Отмена запланированного отчисления

Если до X передумали — нужно снять план. PATCH `/api/subscriptions/[id]` с
`cancelScheduledWithdrawal:true`: очистить `scheduled*`, реактивировать
зачисление (`withdrawnAt=null, isActive=true`). UI — кнопка/бейдж
«Запланировано отчисление на X · Отменить» в карточке абонемента.

## Изменения по файлам

### 1. Схема (`prisma/schema.prisma` + миграция)
`model Subscription` — три новых nullable-поля:
- `scheduledWithdrawalDate DateTime? @map("scheduled_withdrawal_date") @db.Date`
- `scheduledWithdrawalReasonId String? @map("scheduled_withdrawal_reason_id") @db.Uuid` (+ relation на `WithdrawalReason`)
- `scheduledWithdrawalComment String? @map("scheduled_withdrawal_comment")`

Индекс для cron: `@@index([tenantId, scheduledWithdrawalDate])` (или частичный по
`scheduledWithdrawalDate IS NOT NULL` — обычным Prisma-индексом).

### 2. Валидация даты (`lib/subscriptions/last-paid-lesson-date.ts`)
`validateWithdrawalDate` разбить на смысл:
- **прошлое/сегодня** — как сейчас (не раньше `startDate`).
- **будущее** — РАЗРЕШИТЬ, но не позже `endDate`/конца месяца периода (для
  календарного — последний день `periodMonth`). Вернуть флаг/режим
  `immediate | scheduled`, чтобы роут выбрал ветку. Подпись: добавить
  `endDate` в параметры.

### 3. Извлечь общий сеттлмент (`lib/subscriptions/finalize-withdrawal.ts`, новый)
`applyWithdrawalSettlement(tx, { subscription, withdrawalDate, reasonId, comment, createdBy })`:
вся текущая денежная сверка из PATCH/refund (нетто-оплата − отработано − прошлые
сверки → `applyBalanceDelta`, `balance=0`, `status=withdrawn`, `withdrawalDate`,
recalcDiscounts, `churnClientIfNoActiveSubscription`). Переиспользуется в:
немедленном отчислении (PATCH/refund) и cron-финализации. **DRY** — сейчас сверка
дублируется в двух роутах.

### 4. Роут отчисления (`api/subscriptions/[id]/route.ts`, ветка `withdrawn`)
- `effectiveWithdrawalDate <= сегодня` → как сейчас (через `applyWithdrawalSettlement`).
- `> сегодня` → **запланировать**: `deactivateGroupEnrollmentOnWithdrawal`
  (withdrawnAt=X+1), записать `scheduled*`, заметку. НЕ менять `status`/баланс.
- Новый payload-ключ `cancelScheduledWithdrawal` → отмена плана.
- `/refund` POST — аналогичная ветка (на случай вызова не из диалога). Диалог
  ходит через PATCH.

### 5. Cron финализации (новый)
- `lib/cron/finalize-scheduled-withdrawals.ts` — выборка `due` абонементов,
  по каждому в отдельной транзакции `applyWithdrawalSettlement`.
- `api/cron/finalize-scheduled-withdrawals/route.ts` — Bearer `CRON_SECRET`.
- `.github/workflows/finalize-scheduled-withdrawals.yml` — `cron '0 2 * * *'`
  (ежедневно, до прочих в 3:xx).

### 6. Превью (`api/subscriptions/[id]/refund` GET)
Для будущей даты вернуть **прогноз**: `delta_прогноз = нетто-оплачено −
(отработано_факт + цена × занятий_по_расписанию_от_сегодня_до_X)`, плюс флаг
`scheduled:true`. Диалог показывает оценку как «ориентировочно».

### 7. UI диалога (`crm/clients/[id]/client-tabs.tsx`, WithdrawDialog)
- Снять верхнюю границу `max=сегодня` у `input[type=date]` (разрешить будущее в
  пределах периода).
- При будущей дате — пояснение: «Отчисление запланировано на X. Ребёнок ходит до
  X включительно. Возврат/долг рассчитается X числа по факту посещений
  (ориентировочно N ₽)». Кнопка: «Запланировать отчисление на X».
- Карточка абонемента: бейдж «Запланировано отчисление на X · Отменить».

### 8. Справка и доки
- `lib/page-help-content.ts` (ключ `crm/clients/[id]` / абонементы) — описать
  запланированное отчисление.
- `lib/ai-context.ts` FAQ — про отчисление с будущей датой.

## Граничные случаи
- **Дата = сегодня/прошлое** — старое поведение, без изменений.
- **Дата > конца периода** — отклонить (следующий месяц = отдельный абонемент).
- **Отмена до X** — снять `scheduled*`, реактивировать зачисление.
- **Массовое продление (`bulk-renew`)** — исключать источники с
  `scheduledWithdrawalDate != null` (не предлагать продлевать уходящего).
- **Закрытие периода до срабатывания cron** — финализация создаёт балансовую
  проводку «сейчас»; занятия ≤ X уже отмечены. Допустимо.
- **Ребёнок ходил все занятия до X** — `delta = оплачено − всё отработанное`,
  корректно.
- **`status=pending` на момент X** — финализуем так же (сверка по факту).

## Что НЕ делаем
- Не трогаем логику отметки посещений (ни одиночную, ни массовую) — в этом и
  смысл подхода A.
- Не делаем немедленный возврат «наперёд» (это подход B, отклонён).
- Только календарный/фиксированный по периоду; пакетный (`expiresAt`) — вне
  scope.

## Тестирование
- Unit: `validateWithdrawalDate` (будущее в пределах периода / за периодом).
- Unit: `applyWithdrawalSettlement` (возврат/долг/ноль) — вынос из роутов.
- Интеграция/Playwright: запланировать на X → ребёнок виден на занятиях ≤ X,
  отметить часть, дойти cron'ом до X → проверить возврат за непосещённые,
  `status=withdrawn`, клиент в «Выбывших», ребёнок исчез из расписания после X.
- Отмена плана → зачисление вернулось, абонемент снова обычный.
